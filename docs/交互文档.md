> 本文档生成日期：2026-07-08

## 1. 项目定位

EasyBake 是一个 React + TypeScript 的咖啡烘焙管理后台，目前核心围绕单用户工具形态展开，已具备向多用户、权限、工作流、插件系统扩展的基础。

当前交互覆盖以下业务域：

| 业务域     | 当前入口                                  | 主要能力                                            | 当前状态 |
| ---------- | ----------------------------------------- | --------------------------------------------------- | -------- |
| Auth       | `#/login`、`#/register`                   | PocketBase 账号登录、注册、恢复会话、退出登录       | 已接入   |
| Legal      | `#/terms`、`#/privacy`、`#/data-deletion` | 用户协议、隐私政策、数据删除机制公开展示            | 已接入   |
| Bean       | `#/beans`                                 | 生豆库存、创建、搜索、查看、字段级编辑、删除        | 已接入   |
| Roast      | `#/roasts`                                | 烘焙计划列表、手动创建、JSON 导入、查看、编辑、删除 | 已接入   |
| Production | `#/production`                            | 烘焙历史、创建烘焙记录、库存扣减、字段级编辑、删除  | 已接入   |
| Settings   | `#/settings`                              | 熟豆库连接、界面外观、卡片展示、二维码、版本号      | 已接入   |
| Finance    | `#/finance`                               | 经营总览、支出台账、成本模板                        | 已接入   |
| Inventory  | 无独立页面入口                            | 目录预留                                            | 后续扩展 |

## 2. 技术与运行框架

| 层级     | 实现                                 | 交互意义                                                                           |
| -------- | ------------------------------------ | ---------------------------------------------------------------------------------- |
| 应用框架 | React 19 + TypeScript 5 + Vite       | 单页 HashRouter 应用，适合静态部署与后台管理                                       |
| UI 框架  | Ant Design 5                         | 表单、按钮、抽屉、菜单、提示、弹窗、标签等基础交互                                 |
| 路由     | React Router `createHashRouter`      | 使用 `#/path` 路由，刷新页面不依赖服务端路由转发                                   |
| 数据请求 | TanStack Query                       | 列表查询、缓存、乐观更新、失败回滚                                                 |
| 全局状态 | Zustand                              | 鉴权状态、设置状态、布局状态                                                       |
| 表单校验 | AntD Form、React Hook Form、Zod      | 登录注册使用 AntD Form；业务表单使用 schema 校验；所有可编辑控件必须提供可访问名称 |
| 数据源   | PocketBase REST + 熟豆 Supabase REST | 主业务数据走 PocketBase；熟豆镜像和设置页熟豆探活单独使用 Supabase REST            |
| 测试     | Vitest + React Testing Library       | 已覆盖路由、页面、同步、服务、设置等关键交互                                       |

键盘焦点：所有可聚焦控件在 `:focus-visible` 状态下必须显示高对比度黑白灰焦点环；鼠标点击不应触发该焦点环。

时间规则：PocketBase 与前端写入的 `created_at`、`updated_at` 统一保存 UTC ISO 时间，用于排序、同步和计算；用户可见的构建时间、业务时间展示、自动生豆编号与默认采购日期统一按 `Asia/Shanghai`（北京时间）计算。不得通过将数据库 UTC 时间整体加 8 小时的方式伪造本地时间。

## 3. 应用初始化交互

### 3.1 启动顺序

入口链路：

```text
main.tsx
  -> App
    -> AppProviders
    -> AppAuthBootstrap
    -> AppUpdateBanner
    -> AppRealtimeSync
    -> RouterProvider
```

启动时会发生：

1. 清理废弃 localStorage key。
2. 应用壳、认证页与业务页面均由路由按需加载；启动阶段不得预加载未访问的库存、烘焙、生产、财务与设置模块。
3. 初始化 Ant Design 中文配置、黑白灰主题 token、QueryClient、ErrorBoundary。
4. `AppAuthBootstrap` 调用 `useAuthStore.bootstrapSession()` 恢复登录态。
5. `AppRealtimeSync` 在登录后建立 PocketBase realtime 订阅并触发数据刷新。

### 3.2 Provider 交互

`AppProviders` 统一提供：

| 能力                | 行为                                                                                       |
| ------------------- | ------------------------------------------------------------------------------------------ |
| QueryClient         | 默认 `staleTime: Infinity`，避免频繁自动刷新；由手动刷新、实时事件、业务 mutation 主动更新 |
| AntD ConfigProvider | 统一中文、主题、圆角、字体、主色黑白灰                                                     |
| AntD App            | 提供 `message`、`modal` 等反馈 API                                                         |
| ErrorBoundary       | 捕获渲染异常，避免整页崩溃                                                                 |
| 主题同步            | 根据设置页 `themeMode` 写入 `document.documentElement.dataset.appTheme`                    |
| 消息定位            | 根据应用头部高度更新 AntD message top，避免遮挡顶部导航                                    |

### 4.2 鉴权守卫

`RequireAuth`：

1. `hasHydrated === false` 或 `status === 'hydrating'` 时显示全屏 Spin：`正在恢复登录态`。
2. 非 authenticated 时跳转 `#/login`。
3. 已登录时渲染 `MainLayout`。

`PublicOnly`：

1. 恢复登录态期间显示同一 loading。
2. 已登录访问登录/注册页时重定向 `#/beans`。
3. 未登录时渲染登录或注册页。

### 4.3 登录注册交互

登录页字段：

| 字段     | 校验           | 行为                                                     |
| -------- | -------------- | -------------------------------------------------------- |
| 邮箱     | 必填、邮箱格式 | 提交给 `/api/auth/login`；也作为重发验证邮件的目标邮箱   |
| 密码     | 必填           | 提交给 `/api/auth/login`                                 |
| 协议确认 | 必须勾选       | 用户需阅读并同意《用户协议》和《隐私政策》后才能提交登录 |

认证页布局约束：

1. 登录页与注册页的表单块使用同一组纵向定位规则。
2. 默认保持在屏幕中线略靠上位置，避免在移动端落到视口下半区。
3. 无论表单内容后续如何增长，顶部都至少保留 `10vh` 间距。
4. 登录页、注册页、找回密码页顶部统一显示 `EasyBake` 品牌标题。
5. 找回密码页不再展示独立的 `PocketBase Auth / 找回密码 / 描述` 头部文案。
6. 登录页、注册页、找回密码页底部通过统一 `LegalFooter` 展示备案信息：公安联网备案号与图标在上，ICP 备案号在下；备案信息跟随页面自然滚动，不使用固定悬浮底栏。

注册页字段：

| 字段     | 校验               | 行为                                                     |
| -------- | ------------------ | -------------------------------------------------------- |
| 邮箱     | 必填、邮箱格式     | 提交给 `/api/auth/register`                              |
| 密码     | 必填、至少 8 位    | 创建账号后等待邮箱验证                                   |
| 确认密码 | 必填、必须等于密码 | 不一致时阻止提交                                         |
| 协议确认 | 必须勾选           | 用户需阅读并同意《用户协议》和《隐私政策》后才能提交注册 |

注册成功后：

1. 清理本地应用状态。
2. 调用 `/api/auth/register` 创建账号；BFF 仅在该首次创建请求中将注册邮箱写入 `users.name` 作为默认昵称，后续邮箱变化不会覆盖该字段，用户可自行修改昵称。
3. BFF 触发 PocketBase `request-verification`。
4. 邮件中的验证链接指向 EasyBake `#/verify-email?token={TOKEN}`；前端调用同源 `/api/auth/confirm-verification`，由 BFF 再向 PocketBase 提交 token。
5. 验证页成功后引导用户前往登录；无效或过期 token 显示重新发送验证邮件的提示。
6. 不写入登录态，不更新 authenticated 状态。
7. 跳转回 `#/login`，提示用户先完成邮箱验证。

合规公开页面：

1. `#/terms` 展示用户协议。
2. `#/privacy` 展示隐私政策。
3. `#/data-deletion` 展示数据删除机制。
4. 三个页面未登录可访问，顶部提供互相跳转，并展示生效日期、运营主体和联系邮箱。
5. 当前运营主体为个人开发者，合规联系邮箱为 `zhc1501705072@163.com`。
6. 法律页面使用独立纵向滚动容器，避免受全局 `body` 固定视口影响导致长文本无法下滑。
7. 法律页面文章底部提供 `退出` 按钮，点击后回到 `#/login`。
8. 法律页面和应用根容器禁用横向滚动与浏览器横向 overscroll，内容滚动只保留纵向。
9. 法律页面最底部复用 `LegalFooter` 展示备案信息，顺序为公安联网备案号与图标在上、ICP 备案号在下。

设置页账号注销交互：

1. 设置页底部新增 `注销账号` 危险按钮，不再额外展示说明卡片；`当前 Web 上传版本` 展示在该按钮下方。
2. 账号注销按钮默认开放，用户点击 `注销账号` 会弹出确认框。
3. 确认框会明确提示账号、烘焙业务数据、财务记录、库存设置和关联配置会被永久删除且无法恢复。
4. 确认框需要等待 5 秒倒计时结束后，`确认注销` 才可点击。
5. 确认后调用 `/api/auth/account`，删除当前 PocketBase 用户账号及其全部关联业务数据。
6. 删除成功后清空本地登录态、设置缓存、查询缓存，以及当前站点的 `localStorage`、`sessionStorage`、可访问 Cookie、IndexedDB 与 Cache Storage，并回到未登录状态；`HttpOnly` 认证 Cookie 由服务端响应过期。
7. 设置页底部在版本号下方展示公安联网备案号与 ICP 备案号，作为小字号合规信息，不影响设置项布局和危险操作按钮。

登录限制：

1. `/api/auth/login` 成功鉴权后，还会检查 `record.verified`。
2. 若邮箱未验证，则拒绝写入 Cookie，并提示先完成验证。
3. `/api/auth/session` 刷新登录态时也会执行同样的验证检查，避免旧版本自动登录的未验证账号继续进入系统。

辅助认证操作：

1. 登录页支持“重新发送验证邮件”，调用 `/api/auth/request-verification`。
2. `#/forgot-password` 支持发送密码重置邮件，调用 `/api/auth/request-password-reset`；重置邮件跳转至 `#/reset-password?token={TOKEN}`。
3. 重置页要求两次输入至少 8 位且一致的新密码，并调用 `/api/auth/confirm-password-reset`；成功后引导前往登录，无效或过期 token 引导重新发起找回密码。
4. 两个邮件请求接口对邮箱不存在等客户端错误统一返回成功文案，对上游 5xx 仍向前端抛错，便于发现 SMTP 或 PocketBase 异常。

未验证账号清理：

1. BFF 内部任务 `/internal/jobs/cleanup-unverified-users` 仅接受 `127.0.0.1`、`::1` 与 IPv4-mapped loopback 请求，不经 Nginx 向公网暴露。
2. 每日北京时间 `03:00` 由 systemd 分别调用正式 BFF `3001` 和测试 BFF `3002`。
3. 任务只删除 `users` 集合中创建时间超过 24 小时且 `verified = false` 的记录；创建时间无法解析的记录不删除，避免误删。

退出登录：

1. 用户在设置页顶部账号栏点击 `退出登录`。
2. 请求 `/api/auth/logout`，失败时记录 warn 但仍清理本地会话。
3. 清空 QueryClient 缓存、各业务模块运行期内存镜像、Zustand 设置状态，以及当前站点的 `localStorage`、`sessionStorage`、可访问 Cookie、IndexedDB 与 Cache Storage；`HttpOnly` 认证 Cookie 由服务端响应过期。
4. 后续切换到其他账号登录时，首屏不得继续显示上一个账号的生豆、烘焙计划、烘焙历史、财务或设置残留。
5. 跳转 `#/login`。

维护注意：`pocketBaseSessionService` 仅保存当前运行期的公开用户资料，`load/save` 不写入浏览器存储，也不保存 PocketBase Token。会话恢复依赖 `/api/auth/session` 与当前运行环境。

## 5. 主布局交互

### 5.1 桌面端

断点：`md` 及以上。

桌面布局：

```text
左侧 Sider
  - EasyBake 品牌按钮：滚动到顶部
  - 菜单：生豆库存 / 烘焙计划 / 烘焙历史 / 财务 / 设置
  - 业务菜单图标使用未选中/选中双态：生豆库存 `HddOutlined/HddFilled`，烘焙计划 `FundOutlined/FundFilled`，烘焙历史 `FireOutlined/FireFilled`，财务 `AccountBookOutlined/AccountBookFilled`

  - 可缩放内容容器
  - 页面 Outlet
右下悬浮按钮区
  - 快速刷新
  - 页面注册的新增按钮
```

侧边栏折叠状态由 `useAppStore.sidebarCollapsed` 管理。

### 5.2 移动端

移动布局：

```text
顶部 EasyBake 品牌按钮
中间滚动视口
底部 Tab 导航
右下悬浮按钮区
```

移动端交互特性：

| 特性             | 行为                                                                                                                                                                                           |
| ---------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| 底部导航         | 仅保留生豆库存 / 烘焙计划 / 烘焙历史 / 财务四个业务入口，并根据当前路由移动 active pill；图标与桌面侧栏保持同一套未选中/选中双态                                                               |
| 顶部右侧动作     | 当前页面如注册新增动作，则在 Header 右侧显示对应加号入口；设置面板展开时隐藏                                                                                                                   |
| 路由切换动画     | 使用上一个 outlet 与当前 outlet 做左右滑动过渡                                                                                                                                                 |
| 页面滚动复位     | 路由变化时滚动到顶部                                                                                                                                                                           |
| PWA 适配         | 同步安全区、visualViewport、standalone 状态                                                                                                                                                    |
| 触控下拉刷新     | 粗指针设备启用 `GlobalPullToRefresh`；明显横向滑动会优先锁定为横滑手势，不进入下拉刷新状态                                                                                                     |
| 主内容区横滑导航 | 仅在主内容区域生效；左向右滑动切换到左侧一级菜单，位于最左侧菜单时打开设置抽屉；设置抽屉已打开时继续左向右滑不再触发；设置抽屉已打开时右向左滑优先关闭抽屉，普通页面右向左滑切换到右侧一级菜单 |

主内容区横滑导航避让规则：

1. Header、底部导航不作为触发区域；设置抽屉内部仅支持右向左滑关闭设置，左向右滑不触发。
2. 表单输入、按钮、链接、弹窗、抽屉、选择器、Tabs、表格、横向滚动区域不触发全局横滑导航。
3. 复杂组件如图表、横向列表、可拖拽区域可以添加 `data-prevent-swipe-navigation="true"` 主动退出全局横滑接管。
4. 手势必须满足明显横向位移后才触发，避免与纵向滚动、下拉刷新冲突。
5. 当触摸轨迹被判定为横向手势后，下拉刷新在本次触摸周期内保持禁用，避免左滑或右滑时误触发刷新提示。

移动端输入法避让规则：

1. 当可编辑输入框、文本域或会触发软键盘的文本输入控件获得焦点后，当前字段必须保持在可见区域内，不得被系统键盘遮挡。
2. 若字段所在区域存在独立滚动容器，则优先滚动该容器；主页面滚动视口与 `AppDrawer` 内部 `.ant-drawer-body` 都必须应用同一套避让规则。
3. 输入态期间，所有复用 `DrawerActionBar` 的底部 `取消 / 保存` 操作栏需要暂时隐藏，避免与键盘和聚焦字段争抢可见区域；输入态结束后再恢复显示。
4. 当系统键盘收起时，即便输入框已经失焦，也要继续以上一个聚焦字段作为目标，再次回到当前可见滚动区域的中线附近，避免停留在屏幕底边。
5. 若当前滚动区域底部存在 sticky 操作栏，例如抽屉内的 `取消 / 保存` 按钮区，则该操作栏同样视为遮挡区域；输入框聚焦后必须滚动到操作栏上方，而不是被按钮覆盖。

### 5.3 页面动作按钮

页面通过 `ViewportFloatingActionButton` 注册自己的新增动作。

当前注册关系：

| 页面     | 动作文案     | 点击行为                                                         |
| -------- | ------------ | ---------------------------------------------------------------- |
| 生豆库存 | 新增生豆     | 打开创建方式选择，再进入手动创建或 AI 图片识别                   |

### 6.1 快速刷新

触发入口：

| 入口                    | 触发条件           |
| ----------------------- | ------------------ |
| 桌面/非触控悬浮刷新按钮 | 用户点击           |
| 触控端下拉刷新          | 用户下拉           |
| Realtime 事件           | 订阅主题发生变化   |
| 页面可见性恢复          | 页面从隐藏变为可见 |
| 网络恢复                | `online` 事件      |

刷新范围按当前路由决定：

| 当前路由      | 刷新范围                           |
| ------------- | ---------------------------------- |
| `/beans`      | 生豆                               |
| `/roasts`     | 生豆、烘焙计划、烘焙记录           |
| `/production` | 生豆、烘焙记录                     |
| `/finance`    | 生豆、成本核算                     |
| `/settings`   | 设置相关                           |
| all           | 生豆、烘焙计划、烘焙记录、成本核算 |

刷新主流程：

```text
refreshQuickAppData
  -> 清理废弃本地 key
  -> 载入连接配置、成本模板、显示设置
  -> 按当前 scope 同步远端/本地数据
  -> 用远端运行期缓存和正在创建中的乐观临时记录回填 TanStack Query 缓存
  -> 返回 success / failed / downloaded / uploaded
```

生豆同步成功后，`beanCacheService` 中的远端缓存必须作为 `getBootstrappedBeans()` 的数据来源之一。启动同步、下拉刷新和 realtime 刷新回填 Query 时不得只读取本地临时记录，否则会把已从 PocketBase 拉取到的生豆列表覆盖为空，导致生豆页库存概览归零、烘焙计划无法选择生豆。除当前创建请求仍在进行中的 `local-` 乐观记录外，PocketBase 没有的生豆不得继续保留在本地列表中。

设置连接、成本模板或显示设置在快速刷新阶段发生远端异常时，只能记为 `failed > 0` 的部分刷新告警并继续执行当前业务 scope 的数据同步，不得直接向财务页、下拉刷新或桌面刷新按钮冒泡 `CONFIG` 级“主业务数据服务配置异常”提示。

### 6.2 Realtime 订阅

登录且页面可见时创建 `EventSource`：

```text
/api/realtime
```

浏览器不持有 PocketBase Token。实时 SSE 连接和后续订阅请求都只携带同源 `HttpOnly Cookie`；BFF 在服务端校验 Cookie 并向 PocketBase 注入认证信息。业务集合请求同样只允许经 BFF 的白名单路由访问。

订阅主题：

| 主题                            | 影响页面                       |
| ------------------------------- | ------------------------------ |
| `app_settings/*`                | 生豆、烘焙历史、烘焙计划、设置 |
| `bean_sale_specs/*`             | 生豆                           |
| `green_beans/*`                 | 生豆、烘焙历史、烘焙计划       |
| `green_bean_purchase_batches/*` | 生豆                           |
| `roast_batches/*`               | 生豆、烘焙历史、烘焙计划       |
| `roast_profiles/*`              | 烘焙历史、烘焙计划             |

事件刷新有 320ms debounce，避免高频事件导致连续请求。

浏览器刷新、切换页面或网络中断导致 SSE 连接关闭时，BFF 仅中止对应上游 realtime 请求，不得将客户端断开产生的 `AbortError` 冒泡为进程级异常，避免影响后续业务集合读取。

### 6.3 主库认证边界与 BFF 代理

主库请求链路：

```text
Browser
  -> 同源 /api/* 请求（credentials: same-origin）
  -> Nginx
  -> Node BFF
  -> 从 HttpOnly Cookie 读取 PocketBase Token
  -> 127.0.0.1:8090 PocketBase
```

认证交互规则：

1. `/api/auth/login`、`/api/auth/session` 与 `/api/auth/profile` 成功时只返回 `{ record }`；响应不得包含 `token`。
2. BFF 将 PocketBase Token 写入 `HttpOnly; SameSite=Lax; Secure` Cookie。浏览器 JavaScript 不得读取、持久化或在 `Authorization` 请求头中转发该 Token。
3. 前端运行期会话仅保存用户 `id`、`email`、昵称、验证状态等公开资料；刷新页面后必须通过 `/api/auth/session` 恢复。
4. 主业务 `PocketBaseRestClient` 固定访问当前浏览器同源 `/api/collections/*`，即使界面中存在历史主库地址配置，也不得绕过 BFF 直连外部 PocketBase。
5. 外部熟豆镜像是例外：`roastedBean` 配置为 Supabase URL 时，继续使用专用 Supabase 客户端；配置为独立 PocketBase 时需显式关闭主库 BFF 传输，不能复用主库认证 Cookie。

BFF 仅允许以下主业务集合通过浏览器代理访问：

```text
app_settings
bean_sale_specs
cost_calculations
finance_expense_records
green_beans
green_bean_purchase_batches
roast_batches
roast_curve_records
roast_profiles
roast_records
```

实时订阅仅允许 `app_settings/*`、`bean_sale_specs/*`、`green_beans/*`、`green_bean_purchase_batches/*`、`roast_batches/*`、`roast_curve_records/*`、`roast_profiles/*`。不在白名单的集合、方法或主题必须被 BFF 拒绝，不能成为通用代理入口。

当前 BFF 按职责拆分在 `server/auth-bff/`：

| 模块                                                         | 职责                                                         |
| ------------------------------------------------------------ | ------------------------------------------------------------ |
| `router.ts`                                                  | 只负责 `/api/*` 路由分发与方法限制                           |
| `config.ts`、`types.ts`、`utils.ts`                          | 环境变量、共享类型和基础类型守卫                             |
| `http.ts`                                                    | JSON 请求/响应、Cookie、通用 API 成功/失败响应               |
| `pocketbase-client.ts`、`record-utils.ts`                    | PocketBase URL、响应归一化、白名单代理、记录列表与删除工具   |
| `auth-common.ts`、`auth-handlers.ts`                         | 登录、注册、会话刷新、昵称更新、验证邮件、找回密码、退出登录 |
| `collection-handler.ts`、`realtime-handler.ts`               | 主业务集合白名单代理与 realtime SSE/订阅代理                 |
| `ai/handler.ts`、`ai/qiniu-client.ts`、`ai/usage-service.ts` | 生豆 AI 图片识别、七牛 Qwen 调用、额度读取与使用日志         |
| `account-handler.ts`                                         | 账号注销及关联业务数据清理                                   |

BFF 入口 `server/pocketbase-auth-bff.ts` 只允许保留 handler 组装、异常兜底、独立 HTTP 服务启动逻辑；不得再把业务处理器堆回入口文件。

### 6.4 BFF 与 Nginx 发布约束

生产部署必须满足：

1. PocketBase 监听服务器内网地址 `127.0.0.1:8090`；BFF 的 `PB_BASE_URL` 默认值和生产环境显式配置都必须指向该内网地址。前端开发代理的 `VITE_DEV_API_PROXY_TARGET` 是浏览器到 BFF 的入口地址，不能复制为 `PB_BASE_URL`，否则会形成 BFF -> Nginx -> BFF 的代理循环。
2. 一个 systemd 服务的多个 drop-in 不得重复设置 `PB_BASE_URL`；后加载的配置会覆盖前项。更新前应使用 `systemctl` 或 `grep` 确认最终只有一个该变量定义。
3. Nginx 的 `/api/*` 必须整体转发至 `127.0.0.1:3001`，并启用 `proxy_buffering off` 与 `proxy_read_timeout 360s`，以支持 realtime SSE；不得保留将 `/api/collections/*` 直连 `8090` 的 location。
4. BFF 构建产物和前端构建产物需要分别发布。`./deploy.sh` 与 `./deploy_test.sh` 都会先将前端构建结果复制到本机临时快照，再构建 BFF；BFF 候选目录通过远端 `dist/server.next` 路径部署、重启并完成认证探测后，才同步前端快照，避免将 `dist/server` 发布到静态站点目录。测试脚本复用同一发布实现，但固定指向测试 BFF `3002`、测试 PocketBase `8091` 和测试前端版本目录。
5. BFF 更新时，脚本会保留当前 `dist/server` 目录为 `dist/server.previous`。若服务重启或本机 `POST /api/auth/login` 探测不是 `400`，自动恢复 `.previous` 目录并尝试重启旧服务；BFF 验证失败时不得发布前端快照。
6. 前端临时快照目录默认由 `mktemp` 创建，权限可能为 `700`。同步完成后，脚本必须将远端静态站点根目录设为至少 `755`，保证 Nginx 工作进程可穿透目录读取 `index.html` 与 `version.json`。
7. 前端发布采用版本目录：构建产物先上传至 `/var/www/easybake-releases/<version>.next`，确认含有 `index.html` 和 `version.json` 后提升为正式版本目录；`/var/www/easybake` 是 Nginx 固定入口链接，脚本以 `current.next -> current` 的原子重命名切换版本。
8. 切换前脚本把旧入口写入 `/var/www/easybake.previous`。若公网版本号、健康检查或认证探测失败，脚本必须原子切回 `previous`，而不是保留半同步或验收失败的前端版本。

发布验收：

1. `GET /api/health` 返回 `200`。
2. 未携带 Cookie 请求主业务集合返回 `401`。
3. 测试账号登录返回 `200`、设置 Cookie，并且 JSON 响应不含 `token`。
4. 携带该 Cookie 请求主业务集合返回 `200`。
5. 从服务器本机和外部网络各验证一次；若结果不同，应先检查 DNS、云网络入口或旧反向代理实例，不能将本机 curl 成功视为公网发布完成。
6. BFF 必须具备独立 HTTP 契约测试，至少覆盖健康检查、集合与实时请求的未登录拒绝、集合白名单拒绝、认证接口方法限制，以及登录成功时 JSON 不返回 Token、Token 仅写入 `HttpOnly; SameSite=Lax; Secure` Cookie。测试通过真实 Node HTTP 服务调用 BFF handler，上游 PocketBase 必须使用 mock，不能依赖生产服务。

| ----------- | --------------------------------------------------------------------------------------------------------------------------- |

| 取消        | 关闭创建方式选择，不进入创建流程                                                                                            |

AI 图片识别交互规则：

1. 用户在 `AI 图片识别` 抽屉选择 `jpeg / png / webp` 图片，原图前端限制单张不超过 20MB。
2. 前端会先把图片最长边压缩到 1600px，并导出为 2MB 以内的 JPEG Blob；data URL 仅用于本地预览。
3. 前端将压缩后的 JPEG 二进制 body 提交到 `/api/ai/bean-image-recognition`，并显示原图与压缩后的体积；BFF 内部再转换为 data URL 调用七牛云。
4. BFF 使用 HttpOnly Cookie 校验当前用户，前端不接触七牛云 API Key。
5. 创建方式选择打开时，前端先 GET `/api/ai/bean-image-recognition` 读取 `ai_usage_limits` 和当月成功日志，AI 图片识别按钮显示 `本月剩余 x / y`；额度为 0、功能关闭或额度读取失败时禁止点击，读取失败需显示服务端返回的具体原因，便于排查登录态、权限或后台额度字段配置。
6. BFF 读取 `ai_usage_limits` 判断用户是否启用与当月额度；未配置时默认 `10 次/月`。
7. BFF 调用七牛云 Qwen 时传入 `enable_thinking: false` 关闭深度思考，并通过 prompt 约束模型只输出 JSON；随后对七牛云 OpenAI-compatible 返回结构做容错解析；若模型返回可读文本但不是严格 JSON，BFF 会追加一次文本整理请求转成固定字段 JSON，整理请求同样关闭深度思考。只有识别成功、结构化字段解析成功且写入 `success` 日志后才计次，失败写 `failed` 日志但不扣次数。
8. AI 识别产季统一规范为年份后两位，例如 `2026`、`2025/26` 最终回填为 `26`。
9. 识别完成后展示字段预览和当月剩余额度。
10. 点击 `回填表单` 后在同一新增抽屉流程中切到 `新增生豆` 表单，并把识别字段合并到默认生豆表单值；生豆编号仍使用表单自动生成的 `EB-YYMMDDHHmm`，不使用图片识别出的编号覆盖。
11. 识别结果只作为待确认草稿回填到 `BeanForm`，仍需用户确认并走现有表单校验与创建链路。`BeanForm` 只协调 React Hook Form、Zod 校验、成本模板联动和提交；基础信息、烘焙后处理、产地品质、备注及操作栏分别由独立分区组件渲染，所有分区共享同一份 `control/errors`，不得建立第二套表单状态。
12. AI 图片识别抽屉提供 `检测服务`，同源请求 `/api/health`，并在网络失败时展示当前页面地址、健康检查地址、健康检查实测结果与识别接口地址；识别 POST 出现网络失败时会自动补测 `/api/health`，用于区分基础网关不可达和图片识别 POST 链路失败。

### 10.3 新增烘焙记录

点击 FAB 前置检查：

| 条件           | 不满足提示                                       |
| -------------- | ------------------------------------------------ |
| 至少有一条生豆 | 当前还没有可选的生豆，请先创建至少一条生豆数据。 |

创建表单字段：

| 分区     | 字段                                                                   |
| -------- | ---------------------------------------------------------------------- |
| 基本信息 | 烘焙日期、烘焙程度                                                     |
| 生豆选择 | 生豆 Select（必选）                                                    |
| 熟豆名称 | 可选；空则继承生豆名                                                   |
| 去向     | 销售 / 自留；销售时展示本次最终定价                                    |
| 烘焙计划 | 可选；只展示通用计划或当前生豆对应计划                                 |
| 烘焙数据 | 入豆量、出豆量、失水率、发展比、一爆时间、总烘焙时间                   |
| 评价表单 | 综合评分、目标达成度、风味描述、缺陷记录、下次调整建议、训练授权       |
| 备注     | TextArea                                                               |
| 曲线     | HiBean JSON 曲线预选、曲线图、曲线数据摘要；保存成功后绑定到新烘焙记录 |

自动推断：

1. 根据入豆量、出豆量计算脱水率。
2. 用脱水率推断建议烘焙程度。
3. 用户手动改过烘焙程度后，不再自动覆盖。
4. 去向为 `销售` 时，`本次最终定价` 默认取所选生豆的 `最终定价/defaultSaleUnitPrice`，用户可只为本次烘焙记录改写。
5. 评价表单中的 `训练授权` 默认关闭，仅在用户主动勾选时写入授权状态。

曲线导入：

1. 新增烘焙记录、烘焙记录详情和完整编辑抽屉均展示 `曲线` 区块，并复用同一个曲线附件面板，避免创建、编辑、展示维护多套 UI。
2. 新增烘焙记录时点击 `导入 HiBean JSON` 选择 HiBean 导出的 `.json` 文件，前端使用 Zod 校验 `dataList`、`eventList`、`phaseList` 等字段，并立即展示同款曲线预览与曲线数据摘要。
3. 新增记录保存成功后，先拿到真实烘焙记录 ID，再把暂存 JSON 写入 `roast_curve_records` 并同步摘要字段；若保存失败，不创建曲线记录。
4. 已保存记录点击 `导入 JSON` 选择 HiBean 导出的 `.json` 文件，前端使用相同校验与展示组件。
5. 一个烘焙记录只保留一条当前有效曲线；已有曲线时点击 `覆盖导入` 必须二次确认。
6. 导入后写入 `roast_curve_records`，以 `roast_batch_id` 查找已有记录并执行 update 覆盖，不保留历史版本。
7. 曲线图展示 `BT`、`RoR` 和关键事件线；HiBean 设备不支持或恒为 0 的 `ET` 不展示。
8. `RoR` 使用分段控件切换展示模式：`灵敏`、`适中`、`舒缓` 均基于 HiBean 导入的原始 RoR 数据，仅按显示模式做轻量平滑，切换时曲线即时重绘；原始 RoR 小于等于 0 的点不展示，不绘制贴 0 轴的线段。
9. 曲线图采用综合图面展示：在克制浅色图面内同时呈现 `BT`、`RoR`、关键事件标点、事件时间/温度、阶段色条和图例；曲线和关键点允许使用低饱和强调色提高可读性。曲线图下方继续展示除原始曲线点数组外的数据摘要，包括关键指标、事件、阶段、记录来源、设备信息和生豆快照，便于复盘与后续 AI 分析。
10. 导入成功后同步更新烘焙记录摘要字段：`firstCrackTime`、`totalRoastTime`、`developmentRatio`。其中总时长优先取下豆事件，而不是冷却阶段结束时间。
11. 删除烘焙历史必须二次确认，提示关联曲线会永久删除且无法复原；服务端先删除该记录的 `roast_curve_records`，再删除烘焙历史本身。

提交校验：

| 条件         | 失败行为       |
| ------------ | -------------- |
| 烘焙日期为空 | 滚动到日期字段 |
| 未选择生豆   | 滚动到生豆字段 |
| 入豆量 <= 0  | 滚动到入豆量   |
| 出豆量 < 0   | 滚动到出豆量   |

### 10.4 AI 训练准备

烘焙记录详情抽屉与完整编辑抽屉都会展示 `AI 训练准备` 区块，用于提前告知训练上传的必要条件。正式环境仍保持禁用；测试环境先开放点按测试，仅完成服务端校验、训练快照写入和上传审计，不触发真实训练、不生成 AI 推荐、不发布模型版本。

训练就绪项固定为五项：

1. 生豆信息
2. 烘焙计划
3. 目标条件
4. 曲线数据
5. 评价表单
6. 训练授权

交互规则：

1. 每项展示 `已就绪 / 待补充` 状态和缺失说明。
2. 汇总文案根据缺失项自动变化；全部就绪时提示“当前记录已满足后续训练上传条件”。
3. `AI 训练准备` 区块位于烘焙记录详情抽屉与完整编辑抽屉最底部，避免遮挡常用查看和编辑信息。
4. 正式环境底部按钮展示 `上传用于训练（正式环境暂未开放）`，始终禁用。
5. 测试环境底部按钮展示 `上传用于训练`；只有生豆信息、烘焙计划、目标条件、曲线数据、评价表单、训练授权全部满足，且该烘焙记录未上传过时才允许点按。
6. 点按上传前必须弹出确认，提示本次会生成训练快照，上传后用于训练的数据不支持逐条撤回，且同一条烘焙记录不能重复上传。
7. 上传成功后服务端立即对训练样本执行质量检查，并写回 `quality_status`、`quality_report`、`quality_checked_at`。
8. 质量检查通过时 `quality_status = passed`，失败时 `quality_status = failed`；后续训练任务只允许读取 `passed` 样本。
9. 上传成功后按钮变为已上传状态，后续不能重复上传。
10. 训练上传以 PocketBase 数据为准：服务端会重新读取烘焙记录、生豆、烘焙计划、曲线和评价表单，不信任前端缓存状态。
11. 服务端保留 `POST /internal/jobs/check-roast-training-samples` 内部接口，可由服务器本机手动或定时补跑 `pending` 样本的质量检查。

### 10.5 创建记录的数据副作用

创建链路：

```text
ProductionPage.handleCreate
  -> submissionBackupService.save(create)
  -> roastBatchService.createOptimisticBatch
  -> 插入 Query 缓存
  -> roastBatchService.createBatch
  -> 如果新增抽屉暂存了 HiBean JSON
      -> roastCurveService.importHiBeanCurve
      -> useUpdateRoastBatch 同步曲线摘要字段
```

`roastBatchService.createBatch` 内部副作用：

```text
如果状态不是 draft
  -> beanService.adjustRemainingWeight(greenBeanId, inputWeightGrams)
  -> 创建 roast_batches 记录
  -> 如果熟豆库镜像启用
      -> roastedCoffeeBeanMirrorService.syncCreatedBatch
      -> 镜像失败时尝试删除主库 roast batch
  -> 保存本地批次缓存
```

失败回滚：

| 失败位置       | 回滚策略                                            |
| -------------- | --------------------------------------------------- |
| 创建主记录失败 | 回补生豆库存                                        |
| 库存回滚也失败 | 抛出需要人工核对的 AppError                         |
| 熟豆镜像失败   | 尝试删除主记录；删除失败则抛出主库/熟豆库不一致风险 |
| 页面层捕获失败 | 回滚 optimistic batch 并提示                        |

### 10.6 更新与删除记录

更新记录时：

1. 卡片右上角“全部编辑”会打开完整 `RoastBatchDrawer mode="edit"`。
2. 完整编辑抽屉底部统一为“取消 / 保存烘焙记录”，并使用保存图标；按钮组位于内容滚动区内，复用 `DrawerActionBar compact` 的 sticky 样式。
3. 单字段编辑继续使用 `RoastBatchFieldEditorDrawer`，底部统一为“取消 / 保存字段名”。
4. 完整编辑抽屉的内容分区、输入间距与四周留白对齐生豆完整编辑抽屉的样式基线。
5. 新增烘焙记录抽屉与完整编辑抽屉复用同一个 `RoastBatchForm` 可编辑表单组件：分区卡片、标题层级、字段标签、输入圆角、字段校验与底部操作栏均由该组件维护；曲线区域作为插槽传入，分别承载新建时的 JSON 暂存预览和已保存记录的覆盖导入。
6. 曲线导入按钮使用下载图标表达导入本地 JSON。
7. 所有复用共享 `Input` / `Select` 的单行字段都必须保持固定宽度约束：字段容器、输入外壳和触发器统一 `min-width: 0` 与 `max-width: 100%`，超长内容仅在控件内截断显示，不得撑开相邻字段或破坏双列布局。
8. 提交更新前先读取当前 batch。
9. 计算当前库存影响重量与下一状态库存影响重量。
10. 同一生豆只调整差值。
11. 更换生豆时先回补旧生豆，再扣减新生豆。
12. 更新失败时按反向 rollbacks 回滚库存。

删除记录时：

1. 删除入口只保留卡片右上角删除按钮，详情抽屉内不提供删除按钮。
2. 读取当前 batch。
3. 如果记录影响库存，回补生豆库存。
4. 删除 roast_batches。
5. 删除失败时回滚库存调整。

页面 mutation 仍有 Query 缓存层面的乐观移除/恢复。

财务口径定义：

> 2026-07-18 起，以下旧口径由本次定义覆盖：生豆必须选择成本模板；库存先用剩余生豆重量除以模板生豆重量得到可烘焙锅数，再以“模板生豆重量 × (1 - 脱水率)”除以出售单份熟豆重量取每锅计划份数。收入扣除包装、能耗和其他费用，利润再扣除生豆成本与关联邮费；已售份数不得超过本锅计划份数。邮费只能关联去向为销售且已完成的烘焙记录；每条记录的关联份数不得超过其计划份数，同一记录在界面中聚合为一行并通过份数控件调整。邮费按全部关联份数平均分摊，单条记录获得的邮费为其关联份数乘以每份分摊额。

1. `全部花费` 只统计实际支出，不统计摊销口径或核算口径成本。
2. `生豆采购` 不通过手工财务台账录入，而是根据生豆记录中的 `采购日期` 与 `购买总价` 自动计入当前统计范围内的支出。
3. `包装`、`邮费` 与其他经营类支出通过“新增支出”手工录入。
4. `已实现收入` 由烘焙历史销售收入与手工补录收入共同组成：只有去向为“销售”的烘焙记录才计入，按 `已售份数 × 本次最终定价` 计算；旧记录没有已售份数时按 `1` 份兼容，缺少本次最终定价时回退生豆默认单份售价。去向为“自留”的烘焙记录不计入收入。手工补录收入只有状态为“已收款”时计入。
5. 旧烘焙记录默认按“销售”处理，且后续允许在烘焙历史中改成“自留”或重新改回“销售”。
6. 毛利润按“已经卖出的收入 - 生豆采购 - 包装支出 - 邮费支出”理解。
7. 经营利润按“已经卖出的收入 - 当前时间范围内全部实际支出”理解。
8. `当前库存预估收入` 按 `剩余生豆重量(g) ÷ 默认烘焙量(g) × 默认单份售价` 计算；预估成本按 `份数 × 默认烘焙量(g) ÷ 1000 × 当前生豆成本(元/kg)` 计算，预估利润为收入减成本。
9. 已实现利润固定按“已实现收入 - 已售出生豆成本 - 关联邮费”计算，不提供额外成本开关。

新增支出抽屉：

1. 字段只保留 `日期 / 类别 / 金额 / 备注`。
2. 类别顺序为 `包装 / 邮费 / 折旧 / 其他 / 自定义`，其中“自定义”固定放在最底部，并允许创建和复用自定义类别。
3. 自定义类别通过历史支出记录沉淀，后续新增支出时继续可选。
4. 备注字段默认不再展示“可留空”辅助文案，仅在校验失败时显示错误提示。
5. 选择`邮费`后显示“关联烘焙记录”多选框：只提供已完成的销售记录；选择后按记录聚合为单行，并以整数份数控件设置关联份数。份数上限为该锅计划可售份数扣除其他邮费支出已关联的份数；可用份数为零的记录不提供选择。多选摘要只显示关联记录数和总份数，长名称在控件内省略，禁止撑出表单宽度。
6. 新增支出表单与“新增生豆”抽屉统一视觉语言：基础字段为第一组、备注为第二组，分区卡片、字段标签、输入圆角、辅助说明、左右内边距与底部操作栏全部复用同一套黑白灰样式；移动端分区卡片内部 padding 也保持 `16px` 对齐，底部操作栏继续采用“取消 + 主提交”双按钮方案。

补录收入抽屉：

1. 字段只保留 `日期 / 类别 / 金额 / 状态 / 备注`，并复用新增支出的底部抽屉样式；整体分区、字段容器、左右内边距与底部操作栏继续与“新增生豆”抽屉保持一致，底部同样使用“取消 + 主提交”双按钮方案。
2. 类别为 `零售 / 批发 / 其他`，分别对应远端 `retail / wholesale / other`。
3. 状态为 `已收款 / 待收款`；只有已收款收入参与 `已实现收入`、`毛利润` 与 `经营利润` 计算。
4. 保存后写入 `finance_income_records`，标题按类别自动生成，例如 `零售收入`。
5. 手工补录收入会出现在“已实现收入明细”中，并允许在该明细抽屉内直接删除。

财务台账同步补充：

1. 手工支出记录远端集合为 `finance_expense_records`。
2. 手工补录收入远端集合为 `finance_income_records`。
3. 财务页与全局快速刷新会同时同步 `成本核算 + 财务台账`，并把远端最新记录回填到内存态与 Query Cache。
4. 财务成本核算、手工支出与手工补录收入均固定走当前站点同源 `/api/*` 主业务 BFF，不依赖设置页或旧缓存中的 `greenBean.projectUrl`，即使该设置为空也不得提示主业务数据服务配置异常。
5. `finance_income_records` 是补录收入的可选读取依赖；当该集合尚未建立、字段或权限规则不一致、网关异常或读取失败时，财务页读取与下拉同步按空收入处理，不阻断现有支出与成本同步；用户保存补录收入时必须展示 PocketBase 返回的字段、权限或服务错误。
6. 当 PocketBase 未配置、网络不可用或支出远端集合尚未建立时，手工支出会直接报错提示，不再降级写入本地持久化缓存。

烘焙历史补充：

1. 烘焙记录新增 `去向` 字段，选项为 `销售 / 自留`。
2. 新建烘焙记录时默认 `销售`。
3. 去向可在烘焙记录完整编辑与字段级编辑中后续修改。
4. 去向展示与保存以 PocketBase 返回结果为准，不依赖本地 bootstrapped 初值或前端乐观缓存先行覆盖。
5. 去向为 `销售` 时，可编辑 `本次最终定价`；该值只保存到本条烘焙记录，不回写生豆的默认最终定价。
6. 新建烘焙记录时，若去向为 `销售`，创建成功后写入熟豆库的价格使用创建时确定的 `本次最终定价`；后续修改该字段暂不反向同步熟豆库。
7. 去向为 `自留` 时，熟豆镜像价格按 `（生豆价格 / 1000）× 默认烘焙量` 计算，不参与财务页 `已实现收入` 汇总。

生豆采购补充：

1. 生豆创建和编辑时必须维护 `采购日期`。
2. 生豆采购日期作为财务统计中生豆采购支出的归属日期。
3. 当生豆采购日期或购买总价被修改时，财务统计结果会自动同步变化。
4. 删除生豆后，对应生豆采购支出也会随之从统计结果中消失。
5. 移动端采购日期输入框必须限制在字段容器内，不允许原生日期控件边框突出父级卡片或字段边界。

卡片行布局：

1. `UnifiedDataCard` 的左侧字段标签保持单行并垂直居中，固定预留至少 6 个汉字宽度；当标签超过该宽度时以省略号截断，不得换行。
2. 左侧标签与右侧字段值至少保留 `12px` 间距；右侧字段值允许按字段配置换行，且不会压缩左侧标签的预留宽度。

烘焙计划删除同步：

1. 用户确认删除后，计划会同时从 Query 缓存和服务层本地镜像中移除。
2. 删除请求进行期间，列表读取和实时同步必须过滤该计划 ID，禁止旧快照回填或将计划重新上传到 PocketBase。
3. 远端删除成功后结束删除状态；只有远端删除失败时才恢复原计划并显示错误提示。
4. PocketBase 是烘焙计划列表的权威数据源；后台同步仅拉取远端计划，不得把“本地存在、远端不存在”的旧计划自动重新上传。
5. 新计划只能通过用户发起的显式创建或 JSON 导入流程上传，不能由后台同步推断创建。
6. 删除被历史烘焙记录引用的计划时，相关 `roast_batches` 必须同时清空 `roast_plan_id` 与 `roast_plan_name`，避免页面继续显示已删除计划。
7. 删除计划后，烘焙计划列表与烘焙历史列表都需立即反映变更，无需额外刷新。
8. 设置页账号栏在移动端与桌面端均需与下方设置卡片保持同一左右边界。

### 11.2 数据备份与上传

设置页底部在注销账号之前展示 `主动备份` 与 `主动上传` 两个按钮。

交互规则：

1. 点击 `主动备份` 后，前端读取当前登录账号下的业务集合，生成 `easybake.user-data-backup` JSON 文件并触发本地下载。
2. 点击 `主动上传` 后选择 EasyBake 备份 JSON，先根据备份归属账号判断上传方式，再导入到当前登录账号。
3. 备份文件会写入导出账号信息；上传时以前端当前登录账号和备份归属账号判断导入模式。
4. 若备份归属与当前账号相同，必须弹出上传方式选择：`完全与备份同步` 与 `合并备份与账号数据`。
5. `完全与备份同步` 用于把测试端还原为正式端快照：当前账号已存在同 ID 记录时更新为备份内容，备份里有但当前账号没有的记录新增，当前账号有但备份里没有的记录删除。该模式只允许同账号使用，跨账号不得开放。
6. `合并备份与账号数据` 用于保留当前账号已有数据：当前账号已存在同 ID 记录时跳过，只导入备份里多出的记录，当前账号备份外数据保留。
7. 若备份归属与当前账号不同，则固定执行“合并”：导入会丢弃旧备份中的 `id`、`owner`、PocketBase 集合元信息与创建/更新时间，为每条备份记录重新创建当前账号下的新记录。
8. 跨账号合并时不得复用备份记录 ID；每条记录写入成功后必须记录“备份旧 ID → 当前账号新 ID”的映射，并把后续集合里的 `green_bean_id`、`roast_plan_id`、`roast_batch_id`、`bean_id` 与生豆相关设置 key 同步改写。
9. 用户可先在旧账号点击 `主动备份` 下载 JSON，再登录新账号点击 `主动上传`，从而把旧账号数据合并到新账号名下。
10. 上传完成后必须清空 Query 缓存，避免继续展示上传前的旧列表快照。
11. 导出或导入遇到未初始化的可选业务集合时，不得整单失败；该集合导出按 0 条处理，导入按跳过处理，并继续执行其他集合。
12. 导出集合时优先按 PocketBase `created` 排序读取；若目标集合表结构或视图差异导致 HTTP 400，必须自动改为无排序读取。可选集合无排序仍失败时按空集合处理。
13. 同账号导入的重复判断必须以 PocketBase `id` 为准，不得用生豆 `code`、`display_name`、邮箱或其他业务字段替代。跨账号合并时不做旧 ID 去重，所有备份记录都按新 ID 新增；即使两个账号存在相同编号或名称的生豆，也应保留为两条不同生豆，采购批次、销售规格、烘焙计划、烘焙记录、曲线、成本和设置项必须关联到导入后映射出的对应生豆 ID。
14. 所有业务集合必须共用同一导入策略，不能为单个集合私自保留旧 `id` 或绕过账号模式判断。同账号补充时，如果按旧 ID 查询已存在则按用户选择跳过或更新；如果按旧 ID 查询不存在但插入阶段返回 `id` 已存在或唯一冲突，必须去掉旧 `id` 后重新创建当前账号记录，并登记旧 ID 到新 ID 的映射，避免 `app_settings` 中某条旧 ID 冲突阻断或跳过 `cost_template_settings` 等后续成本模板数据。
15. 可选集合的 HTTP 400 容错只允许用于读取阶段；写入阶段只有真正的集合不存在/未初始化才可跳过。`app_settings` 写入时若是普通字段校验失败，必须暴露错误；若缺失的是 `cost_template_settings` 等设置记录，则必须按同账号补充规则插入，且导入 payload 不携带 `created_at`、`updated_at` 等时间元字段。
16. 上传完成提示必须展示新增、更新、删除、跳过数量，便于用户判断完全同步是否真的覆盖了测试端旧数据。

1. 写入 `appDisplaySettings`。
2. `UnifiedDataCard` 通过 `useVisibleCardMetaItems` 读取配置。
3. 卡片立即响应展示变更。

烘焙记录扣减库存规则：

1. 创建非草稿烘焙记录时，会按入豆量扣减关联生豆的剩余库存。
2. 当剩余库存与本次入豆量完全相等时，允许创建成功，并将剩余库存扣减为 `0`。
3. 库存比较与写回统一按整数克处理，避免浮点误差导致“剩余 200g、入豆 200g”这类临界值被误判为库存不足。

共享输入控件规则：

1. 普通单行输入框、密码输入框与多行备注输入框统一复用共享 `ui/input` 原语。
2. 数字输入统一复用共享 `ui/input-number` 原语，不再直接依赖旧的第三方数值输入控件。
3. 普通单选下拉统一复用共享 `ui/select` 原语，不再直接使用 Ant Design `Select`；仅 `tags / 自由输入标签` 这类需要多值录入的特例，允许继续保留专用实现。
4. 数字输入在编辑过程中允许用户先完整删空旧值，再输入新值；只有当输入框失焦且仍为空时，才恢复到上一次有效值，不能在删除到空的瞬间立即自动回填。
5. 所有“日期 / 日期时间”字段在粗指针移动端统一复用共享 `AdaptiveDateTimeField`：点击字段后只允许从底部抽屉选择 `年 / 月 / 日 / 时 / 分`，不得直接唤起文本输入框或系统键盘；桌面端继续保留现有日历选择器交互。

### 11.7 二维码与版本

二维码按钮：

| 按钮           | 行为                         |
| -------------- | ---------------------------- |
| 和作者交流一下 | 展示作者交流二维码           |
| 进群交流 bugs  | 展示服务器托管的微信群二维码 |
| 请作者喝杯咖啡 | 展示赞助二维码               |

首次点击时才按需加载对应二维码资源；加载期间保留固定方形占位，成功后展示二维码。同一打包二维码在加载进行中被连续点击或被多个界面同时请求时，合并为一次资源导入。加载失败时展示错误状态和“重新加载二维码”入口，失败请求不保留缓存，以允许重新加载；再次点击同一按钮会收起二维码，后续再次打开复用已加载资源。群二维码在每次打开时追加时间参数重新请求服务器资源，确保更新后无需重新发布前端。

作者交流与赞助二维码的唯一权威资源位于 `src/assets/settings-codes`，由 Vite 生成带内容指纹的按需资源；前端按需加载与 PNG 原图像素一致的无损 WebP 版本。若浏览器无法解码 WebP，图片错误事件才会按需加载 PNG 回退版本；`public` 目录不得保留重复副本。群二维码唯一位于服务器 `/var/www/easybake-assets/community-qr.png`，正式站与测试站均通过 Nginx 的 `/community-qr.png` 读取；本机执行 `./update_community_qr.sh <image-path>` 即可原子替换，且不触发前端或 BFF 发布。脚本接受 PNG 和 JPEG，并在本机将 JPEG 转换为服务器统一使用的 PNG。

版本号来自 `useAppBuildVersion`。前端构建版本号统一展示为纯数字：`package.json` 版本号去除非数字字符后拼接北京时间 `yyyyMMddHHmmss`，例如 `0.1.0` 在 `2026-07-15 16:28:23` 发布时展示为 `01020260715162823`；旧的 `0.1.0-ISO时间` 运行期版本也需按同一规则兼容展示。

### 11.8 主业务数据服务与缓存处理

主业务数据请求固定通过当前站点同源的 `/api/*` 进入 BFF，不提供普通用户可修改的主库地址。设置页只允许配置熟豆镜像的外部服务；主业务数据缓存由登录状态、写入结果和 realtime 事件统一失效或刷新。

主业务数据服务不可用时，页面不得引导用户创建或连接“生豆数据库”；应提示刷新、检查网络、重新登录或联系管理员。

### 12.2 Repository 选择策略

生豆、烘焙计划、烘焙记录服务层都遵循类似策略：

```text
test 环境
  -> mock/local repository
未配置连接
  -> mock/local repository
已配置连接
  -> REST repository
```

财务模块当前拆分为两类持久化路径：

```text
成本核算
  -> PocketBase 远端优先
  -> 当前运行期内存态回填
  -> 快速刷新时重新拉取远端

手工收入 / 手工支出
  -> PocketBase 远端优先
  -> 当前运行期内存态回填
  -> 快速刷新时重新拉取远端
```

注意：主业务仓储默认使用 PocketBase 配置与 `PocketBaseRestClient`；只有熟豆镜像与设置页熟豆探活在 `roastedBean` 为 Supabase URL 时使用 `RoastedBeanSupabaseDataClient`。

### 12.3 乐观更新模式

常见写入模式：

```text
onMutate
  -> cancelQueries
  -> 保存 previous cache
  -> 写入 optimistic cache
mutationFn
  -> 调用 service / repository
onError
  -> 恢复 previous cache
  -> 恢复本地 optimistic snapshot
onSuccess
  -> 用服务端返回值替换缓存
```

页面层额外使用：

```text
submissionBackupService.save(action, payload, domain)
```

用于在失败或异步同步前保留提交内容。

### 12.4 PocketBase 权威同步

主业务数据现统一以 PocketBase 为准：

| 场景                            | 行为                                   |
| ------------------------------- | -------------------------------------- |
| PocketBase 有，本地没有         | 拉取远端并补入本地运行期缓存           |
| PocketBase 没有，本地还有旧记录 | 从本地运行期缓存删除，不得自动回传远端 |
| 创建请求仍在进行中              | 允许短暂展示 `local-` 乐观临时记录     |
| 创建失败 / 更新失败 / 删除失败  | 回滚乐观状态并提示，不保留待同步队列   |

补充约束：

```text
roastBatchService.syncLocalAndRemote()
roastPlanService.syncLocalAndRemote()
  -> 只同步真实远端记录
  -> 跳过 id 以 local- 开头的乐观临时记录
  -> 刷新完成后继续保留这些乐观记录，等待原始创建请求完成

beanService.syncLocalAndRemote()
  -> 只以 PocketBase 返回结果更新运行期缓存
  -> 清理未处于创建中的本地残留
  -> 不再把本地旧记录反向上传到 PocketBase
```

### 12.5 熟豆镜像

启用条件：

```text
roastedBean connection 已配置
非 test 环境
```

镜像目标：

```text
coffee_beans
```

镜像数据包含：

```text
烘焙记录字段
生豆来源字段
默认售价/规格
拼配组件
熟豆展示字段
状态、图片、评分预留字段
```

名称规则：

```text
优先使用：熟豆名称 + 空格 + 处理法
若熟豆名称为空：生豆名称 + 空格 + 处理法
若处理法为空：仅保留可用名称本身
```

烘焙商规则：

```text
优先读取当前登录用户昵称
即 PocketBase users.name
若昵称为空，则 roaster 保持空字符串
```

风味规则：

```text
生豆主档在 PocketBase 里使用 flavor_tags 单字段存储
格式为逗号分隔字符串，例如：柑橘,花香,莓果
前端读取后解析成 string[] 标签数组
创建烘焙历史同步到熟豆 Supabase 时，写入 flavor: string[]
例如：["柑橘","花香","莓果"]
```

烘焙后处理规则：

```text
生豆主档使用 aging_days / tasting_end_days 存储
创建烘焙历史同步到熟豆 Supabase 时：
startDay = agingDays
endDay = tastingEndDays
默认值分别为 14 / 40
```

用途：把 EasyBake 生产出的熟豆记录同步到熟豆展示或销售系统。

## 17. 当前实现的维护注意点

1. 熟豆 Supabase 专属能力集中在 `RoastedBeanSupabaseDataClient`、`roastedBeanSupabaseConnectionSyncService` 与 `roastedCoffeeBeanMirrorService`；除这两条链路外，主业务数据默认按 PocketBase 理解。
2. `pocketBaseSessionService` 是只含公开用户资料的内存会话，不是持久 localStorage token。刷新后的登录恢复、业务集合访问与 realtime 订阅均通过 HttpOnly Cookie + BFF 完成，前端不得再保存或转发 PocketBase Token。
3. Settings 的连接配置服务目前也是内存状态服务，远端同步通过 app settings 补齐。若做离线持久化，需要谨慎设计迁移和清理策略。
4. 烘焙记录创建、更新、删除会影响生豆库存，是高风险交互；调整前必须补充库存回滚测试。
5. 熟豆镜像失败时有回滚主记录逻辑；如果未来改成最终一致性，需要明确用户可见状态和补偿任务。
6. Finance 已落独立交互页面，但真实销售订单与财务自动汇总仍未接入；当前收入、经营费用属于前端本地台账，后续扩展时应继续保持领域拆分，不要把业务逻辑塞回现有页面。
7. 页面文件中已有一些中文注释和业务流程注释，新增代码仍应保持组件单一职责，避免页面继续膨胀。`BeanPage` 只协调查询、列表、详情编辑与乐观写入；新增方式选择、AI 额度读取和创建抽屉由 `BeanCreationFlow` 维护。`SettingsPage` 只协调远端连接初始化、二维码支持与账号注销；外观和卡片展示设置由 `AppearanceSettingsSection` 维护。页面与组件文件不得超过 500 行。
8. AI 图片识别只负责生成待确认草稿，不直接创建生豆；后续扩展多图识别或字段置信度时，仍应保留用户确认步骤。
9. `deploy.sh` 会发布正式前端和 BFF；`deploy_test.sh` 通过环境参数复用同一实现并发布测试环境。BFF 目标、远端入口文件路径、构建目录、服务名、本机探测端口和 SSH 主机可分别通过 `BFF_REMOTE_TARGET`、`BFF_REMOTE_PATH`、`BFF_REMOTE_DIR`、`BFF_SERVICE_NAME`、`BFF_LOCAL_PORT`、`REMOTE_SSH_TARGET` 覆盖；前端版本目录、入口链接和发布 SSH 主机可通过 `FRONTEND_RELEASES_PATH`、`FRONTEND_CURRENT_LINK`、`FRONTEND_REMOTE_SSH_TARGET` 覆盖。测试站当前不再默认使用 Basic Auth，便于验证登录态、PWA 和 AI 训练上传接口。发布脚本会拒绝带有密钥语义的 `VITE_*` 环境变量，并在上传前扫描前端快照，发现 `.env*`、证书私钥、部署密码或服务端密钥标记时必须中止发布。
10. 前端发布目录由 `FRONTEND_RELEASES_TO_KEEP` 控制，默认总数为 `5`，且不得小于 `2`。当前入口和上一版本入口指向的目录永久受保护；公网验收通过后，脚本仅删除超出该总数的最旧非保护版本。当前入口切换、回退入口以及清理均以服务器符号链接解析结果为准，避免删除可回退版本。
11. 发布流程在部署 BFF 前必须获取服务器范围的目录锁，默认路径为 `${FRONTEND_RELEASES_PATH}/.easybake-deploy.lock`。锁由原子创建目录获得，保存本次发布的 owner、创建时间、最近更新时间和执行阶段；锁已存在时后发任务必须立即退出并展示这些元数据，不能上传或切换任何 BFF、前端版本或入口。BFF 部署、前端发布、前端清理三个阶段均刷新锁状态。成功、失败和异常退出均只释放 owner 匹配的锁；前端版本清理必须排除隐藏锁目录。发布客户端强制终止后，必须人工确认任务停止并读取锁元数据，才可删除该精确锁目录；不得自动按超时抢占。
12. PWA manifest 使用 `crossorigin="use-credentials"`，使受 Nginx Basic Auth 保护的测试环境能够携带凭据读取 `site.webmanifest`；正式同源环境同样保持有效。PocketBase 的 `updated_at` 进入显示设置内存状态前必须转换为标准 ISO 时间，不能把数据库原始时间字符串直接写入严格 schema 状态。
13. 主布局拆分为编排层和展示层：`MainLayout` 只保留路由动画、移动端设置浮层时序、浮动按钮注册、滑动导航和刷新/PWA 状态编排；桌面侧栏、移动顶栏、移动底栏、设置浮层、认证栏、浮动按钮容器和路由面板渲染均放在 `src/layouts/components/`。后续调整导航或设置账号栏时优先修改对应小组件，避免把 JSX 和业务处理重新塞回主布局。
14. 黑白灰规范下，持续可见的结构性强调不再使用绿、蓝、橙、红色条或偏色描边，统一改为灰阶层级表达；只有真正的危险操作与系统级错误可继续使用 danger 语义色。下拉刷新、模块占位页统计卡片、底部导航激活态均已按此规则收口。
15. #28～#33 已完成巨型文件收口：`bean.service.ts`、`roastPlan.service.ts`、`roastBatch.service.ts`、`pocketBaseRestClientCore.ts`、`BeanAiRecognitionPlaceholder.tsx`、`CostTemplateManagerPanel.tsx` 现在都只保留门面编排；后续新增逻辑应优先落到各自的拆分目录（如 `bean-service/`、`roast-plan/`、`roast-batch/`、`pocketbase-rest/`、`bean-ai-recognition/`、`cost-template-manager/`），不要把实现重新堆回主文件。
16. 主布局样式也已按组件归属拆分：壳层滚动与缩放样式集中在 `src/layouts/MainLayoutShell.module.css`；桌面导航、移动头部、底部导航、设置浮层、认证栏、浮动按钮和路由转场各自维护独立 CSS module。后续样式调整优先修改对应组件旁的样式文件，不再回到单一总样式表。
17. 生豆仓储实现已继续细分：`bean.service.repositories.ts` 现在只负责仓储解析与导出，Mock 仓储、简单远端仓储、PocketBase 库存仓储以及库存仓储工具分别落在 `bean.service.mock-repository.ts`、`bean.service.remote-repository.ts`、`bean.service.inventory-repository.ts`、`bean.service.inventory-repository.utils.ts`。后续新增集合读写、设置聚合或 payload 转换时，应优先落到对应子文件，避免再次把全部逻辑堆回单个 repositories 文件。
18. `settingsPage.test.tsx` 已保留为主断言文件，但浏览器环境重置、连接卡片展开与运行时 mock 初始化已经抽到 `src/tests/settingsPage.test.shared.tsx`。后续新增设置页测试时，优先复用共享基座，避免测试文件再次膨胀和重复维护。
19. 测试基线已针对当前沙箱环境收口：`server/pocketbase-auth-bff.test.ts` 不再通过真实端口监听验证网关，而是直接调用 handler 完成契约断言；`beanPage.test.tsx` 则统一在数据可见后再执行交互，避免依赖 React Query 的异步刷新时序。后续补测试时，优先保持这种“少依赖环境、少依赖竞态”的测试方式。
