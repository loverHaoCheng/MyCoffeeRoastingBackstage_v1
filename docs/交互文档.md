> 本文档生成日期：2026-07-08

## 1. 项目定位

EasyBake 是一个 React + TypeScript 的咖啡烘焙管理后台，目前核心围绕单用户工具形态展开，已具备向多用户、权限、工作流、插件系统扩展的基础。

当前交互覆盖以下业务域：

| 业务域 | 当前入口 | 主要能力 | 当前状态 |
| --- | --- | --- | --- |
| Auth | `#/login`、`#/register` | PocketBase 账号登录、注册、恢复会话、退出登录 | 已接入 |
| Bean | `#/beans` | 生豆库存、创建、搜索、查看、字段级编辑、删除、待同步本地缓存 | 已接入 |
| Roast | `#/roasts` | 烘焙计划列表、手动创建、JSON 导入、查看、编辑、删除 | 已接入 |
| Production | `#/production` | 烘焙历史、创建烘焙记录、库存扣减、字段级编辑、删除 | 已接入 |
| Settings | `#/settings` | 熟豆库连接、界面外观、卡片展示、二维码、版本号 | 已接入 |
| Finance | `#/finance` | 经营总览、支出台账、成本模板 | 已接入 |
| Inventory | 无独立页面入口 | 目录预留 | 后续扩展 |

## 2. 技术与运行框架

| 层级 | 实现 | 交互意义 |
| --- | --- | --- |
| 应用框架 | React 19 + TypeScript 5 + Vite | 单页 HashRouter 应用，适合静态部署与后台管理 |
| UI 框架 | Ant Design 5 | 表单、按钮、抽屉、菜单、提示、弹窗、标签等基础交互 |
| 路由 | React Router `createHashRouter` | 使用 `#/path` 路由，刷新页面不依赖服务端路由转发 |
| 数据请求 | TanStack Query | 列表查询、缓存、乐观更新、失败回滚 |
| 全局状态 | Zustand | 鉴权状态、设置状态、布局状态 |
| 表单校验 | AntD Form、React Hook Form、Zod | 登录注册使用 AntD Form；业务表单使用 schema 校验 |
| 数据源 | PocketBase REST + 熟豆 Supabase REST | 主业务数据走 PocketBase；熟豆镜像和设置页熟豆探活单独使用 Supabase REST |
| 测试 | Vitest + React Testing Library | 已覆盖路由、页面、同步、服务、设置等关键交互 |

## 3. 应用初始化交互

### 3.1 启动顺序

入口链路：

```text
main.tsx
  -> App
    -> AppProviders
    -> AppAuthBootstrap
    -> AppUpdateBanner
    -> AppRealtimeSync
    -> RouterProvider
```

启动时会发生：

1. 清理废弃 localStorage key。
2. 空闲时预加载 auth、bean、production、roast、finance、settings 模块。
3. 初始化 Ant Design 中文配置、黑白灰主题 token、QueryClient、ErrorBoundary。
4. `AppAuthBootstrap` 调用 `useAuthStore.bootstrapSession()` 恢复登录态。
5. `AppRealtimeSync` 在登录后建立 PocketBase realtime 订阅并触发数据刷新。

### 3.2 Provider 交互

`AppProviders` 统一提供：

| 能力 | 行为 |
| --- | --- |
| QueryClient | 默认 `staleTime: Infinity`，避免频繁自动刷新；由手动刷新、实时事件、业务 mutation 主动更新 |
| AntD ConfigProvider | 统一中文、主题、圆角、字体、主色黑白灰 |
| AntD App | 提供 `message`、`modal` 等反馈 API |
| ErrorBoundary | 捕获渲染异常，避免整页崩溃 |
| 主题同步 | 根据设置页 `themeMode` 写入 `document.documentElement.dataset.appTheme` |
| 消息定位 | 根据应用头部高度更新 AntD message top，避免遮挡顶部导航 |

## 4. 路由与鉴权

### 4.1 路由表

| 路由 | 页面 | 访问条件 | 备注 |
| --- | --- | --- | --- |
| `#/` | 重定向到 `#/beans` | 已登录 | 默认首页 |
| `#/beans` | `BeanPage` | 已登录 | 生豆库存 |
| `#/roasts` | `RoastPage` | 已登录 | 烘焙计划 |
| `#/production` | `ProductionPage` | 已登录 | 烘焙历史 |
| `#/finance` | `FinancePage` | 已登录 | 财务 |
| `#/settings` | `SettingsPage` | 已登录 | 设置 |
| `#/login` | `LoginPage` | 未登录 | 已登录则跳到 `#/beans` |
| `#/register` | `RegisterPage` | 未登录 | 已登录则跳到 `#/beans` |
| `#/forgot-password` | `ForgotPasswordPage` | 任意 | 发送密码重置邮件 |
| `*` | 重定向到 `#/` | 任意 | 兜底 |

### 4.2 鉴权守卫

`RequireAuth`：

1. `hasHydrated === false` 或 `status === 'hydrating'` 时显示全屏 Spin：`正在恢复登录态`。
2. 非 authenticated 时跳转 `#/login`。
3. 已登录时渲染 `MainLayout`。

`PublicOnly`：

1. 恢复登录态期间显示同一 loading。
2. 已登录访问登录/注册页时重定向 `#/beans`。
3. 未登录时渲染登录或注册页。

### 4.3 登录注册交互

登录页字段：

| 字段 | 校验 | 行为 |
| --- | --- | --- |
| 邮箱 | 必填、邮箱格式 | 提交给 `/api/auth/login`；也作为重发验证邮件的目标邮箱 |
| 密码 | 必填 | 提交给 `/api/auth/login` |

认证页布局约束：

1. 登录页与注册页的表单块使用同一组纵向定位规则。
2. 默认保持在屏幕中线略靠上位置，避免在移动端落到视口下半区。
3. 无论表单内容后续如何增长，顶部都至少保留 `10vh` 间距。
4. 登录页、注册页、找回密码页顶部统一显示 `EasyBake` 品牌标题。
5. 找回密码页不再展示独立的 `PocketBase Auth / 找回密码 / 描述` 头部文案。

注册页字段：

| 字段 | 校验 | 行为 |
| --- | --- | --- |
| 邮箱 | 必填、邮箱格式 | 提交给 `/api/auth/register` |
| 密码 | 必填、至少 8 位 | 创建账号后等待邮箱验证 |
| 确认密码 | 必填、必须等于密码 | 不一致时阻止提交 |

注册成功后：

1. 清理本地应用状态。
2. 调用 `/api/auth/register` 创建账号。
3. BFF 触发 PocketBase `request-verification`。
4. 不写入登录态，不更新 authenticated 状态。
5. 跳转回 `#/login`，提示用户先完成邮箱验证。

登录限制：

1. `/api/auth/login` 成功鉴权后，还会检查 `record.verified`。
2. 若邮箱未验证，则拒绝写入 Cookie，并提示先完成验证。
3. `/api/auth/session` 刷新登录态时也会执行同样的验证检查，避免旧版本自动登录的未验证账号继续进入系统。

辅助认证操作：

1. 登录页支持“重新发送验证邮件”，调用 `/api/auth/request-verification`。
2. `#/forgot-password` 支持发送密码重置邮件，调用 `/api/auth/request-password-reset`。
3. 两个接口对邮箱不存在等客户端错误统一返回成功文案，对上游 5xx 仍向前端抛错，便于发现 SMTP 或 PocketBase 异常。

退出登录：

1. 用户在设置页顶部账号栏点击 `退出登录`。
2. 请求 `/api/auth/logout`，失败时记录 warn 但仍清理本地会话。
3. 清空 QueryClient 缓存。
4. 跳转 `#/login`。

维护注意：`pocketBaseSessionService` 当前是内存会话，虽然暴露 storage key，但 `load/save` 未写入浏览器存储。会话恢复依赖 `/api/auth/session` 与当前运行环境。

页面通过 `ViewportFloatingActionButton` 注册自己的新增动作。

当前注册关系：

| 页面 | 动作文案 | 点击行为 |
| --- | --- | --- |
| 生豆库存 | 新增生豆 | 打开新增生豆抽屉 |
| 烘焙计划 | 新增烘焙计划 | 打开新增计划抽屉 |

## 6. 全局刷新与实时同步

### 6.1 快速刷新

触发入口：

| 入口 | 触发条件 |
| --- | --- |
| 桌面/非触控悬浮刷新按钮 | 用户点击 |
| 触控端下拉刷新 | 用户下拉 |
| Realtime 事件 | 订阅主题发生变化 |
| 页面可见性恢复 | 页面从隐藏变为可见 |
| 网络恢复 | `online` 事件 |

刷新范围按当前路由决定：

| 当前路由 | 刷新范围 |
| --- | --- |
| `/beans` | 生豆 |
| `/roasts` | 生豆、烘焙计划、烘焙记录 |
| `/production` | 生豆、烘焙记录 |
| `/finance` | 生豆、成本核算 |
| `/settings` | 设置相关 |
| all | 生豆、烘焙计划、烘焙记录、成本核算 |

刷新主流程：

```text
refreshQuickAppData
  -> 清理废弃本地 key
  -> 载入连接配置、成本模板、显示设置
  -> 同步生豆待处理操作
  -> 按当前 scope 同步远端/本地数据
  -> 回填 TanStack Query 缓存
  -> 返回 success / failed / downloaded / uploaded
```

### 6.2 Realtime 订阅

登录且页面可见时创建 `EventSource`：

```text
/api/realtime
```

订阅主题：

| 主题 | 影响页面 |
| --- | --- |
| `app_settings/*` | 生豆、烘焙历史、烘焙计划、设置 |
| `bean_sale_specs/*` | 生豆 |
| `green_beans/*` | 生豆、烘焙历史、烘焙计划 |
| `green_bean_purchase_batches/*` | 生豆 |
| `roast_batches/*` | 生豆、烘焙历史、烘焙计划 |
| `roast_profiles/*` | 烘焙历史、烘焙计划 |

事件刷新有 320ms debounce，避免高频事件导致连续请求。

### 8.2 搜索与列表

预览字段由设置页 `beanInventory` 卡片展示配置决定。`local-` 开头的本地草稿会展示所有字段，帮助用户识别待同步数据。

卡片头部交互：

2. 右侧操作按钮顺序为查看、全部编辑、删除。
3. 点击“全部编辑”直接打开完整生豆表单抽屉。
4. 点击字段行仍保留单字段快捷编辑能力。
5. “成本”字段为只读计算值，按 `采购总价 / 采购总重 × 1000` 自动得出，不提供单字段修改入口。

### 8.4 新增生豆

点击 FAB 前置检查：

1. 生豆数据库连接必须存在。
2. 至少有一个成本模板。

不满足时提示：

| 条件 | 提示 |
| --- | --- |
| 未配置生豆连接 | 请先前往设置页创建并连接生豆数据库，完成后才能新增数据。 |
| 无成本模板 | 请先前往财务页创建至少一个成本模板，再新增生豆。 |

新增抽屉包含两个 Tab：

| Tab | 状态 |
| --- | --- |
| 界面创建 | 已实现，使用 `BeanManualCreator` + `BeanForm` |
| AI 图片识别 | 占位展示，后续扩展 |

默认表单值：

```text
默认烘焙量 200g
采购总重 1000g
剩余重量 1000g
风味标签默认空数组
其他文本字段为空
成本模板默认自动带入
```

基础信息新增字段：

```text
风味
  - 自由输入
  - 按 Enter 生成标签
  - 支持中英文逗号自动拆分
  - UI 以多个独立 chip 展示
  - 后台以单字段 `flavor_tags`，用逗号分隔保存
```

烘焙后处理新增分组：

```text
养豆时间
  - 数字，单位天
  - 默认 14
赏味结束期
  - 数字，单位天
  - 默认 40
  - 同步熟豆镜像时复用为 endDay
```

提交链路：

```text
BeanManualCreator
  -> BeanForm 校验
  -> BeanPage.handleCreateBean
  -> submissionBackupService.save(create)
  -> beanService.createOptimisticBean
  -> queryClient.setQueryData 插入乐观记录
  -> beanService.createRemoteBean
    -> 成功：finalizeOptimisticBean，替换为远端记录
    -> 网络/404：persistOptimisticBeanAsPending，本地待同步
    -> 其他错误：rollbackOptimisticBean，回滚并提示
```

### 8.6 字段级编辑

点击卡片字段行：

1. 根据字段 key 映射到 `GreenBeanFormInput` fieldPath。
2. 打开 `BeanFieldEditorDrawer`。
3. 抽屉先用卡片数据生成 fallback draft。
4. 同时请求 `beanService.getEditableBean` 获取完整可编辑详情。
5. 用户已编辑时不会被异步详情覆盖。

字段控件类型：

| 字段类型 | 控件 |
| --- | --- |
| 重量、价格、海拔、密度、含水率 | `InputNumber` |
| 养豆时间、赏味结束期 | `InputNumber` |
| 成本模板 | `Select` |
| 风味 | `Select mode="tags"` |
| 备注 | `Input.TextArea` |
| 文本字段 | `Input` |

提交链路：

```text
Zod schema 校验
  -> 失败：提示第一条错误并滚动到字段
  -> 成功：关闭抽屉
      -> submissionBackupService.save(update)
      -> useUpdateBean.mutateAsync
      -> onMutate 乐观更新列表和详情缓存
      -> onError 回滚
      -> onSuccess 写入远端返回值
```

### 8.7 删除生豆

点击删除：

1. `modal.confirm` 二次确认。
2. 确认文案提示会一并删除关联采购批次、烘焙记录等数据。
3. `useDeleteBean` 先从 Query 缓存移除。
4. `beanService.prepareOptimisticDelete` 保存删除快照。
5. 成功后清理详情缓存并 invalidate 列表。
6. 失败时恢复列表、详情与本地快照。

```text

  -> 插入 Query 缓存
  -> roastBatchService.createBatch
```

`roastBatchService.createBatch` 内部副作用：

```text

  -> beanService.adjustRemainingWeight(greenBeanId, inputWeightGrams)
  -> 创建 roast_batches 记录
  -> 如果熟豆库镜像启用
      -> roastedCoffeeBeanMirrorService.syncCreatedBatch

  -> 保存本地批次缓存
```

失败回滚：

| 失败位置 | 回滚策略 |
| --- | --- |
| 创建主记录失败 | 回补生豆库存 |

### 10.5 更新与删除记录

更新记录时：

1. 卡片右上角“全部编辑”会打开完整 `RoastBatchDrawer mode="edit"`。
2. 完整编辑抽屉底部统一为“取消 / 保存烘焙记录”，并使用保存图标；按钮组位于内容滚动区内，复用 `DrawerActionBar compact` 的 sticky 样式。
3. 单字段编辑继续使用 `RoastBatchFieldEditorDrawer`，底部统一为“取消 / 保存字段名”。
4. 完整编辑抽屉的内容分区、输入间距与四周留白对齐生豆完整编辑抽屉的样式基线。
5. 提交更新前先读取当前 batch。
6. 计算当前库存影响重量与下一状态库存影响重量。
7. 同一生豆只调整差值。
8. 更换生豆时先回补旧生豆，再扣减新生豆。
9. 更新失败时按反向 rollbacks 回滚库存。

删除记录时：

1. 删除入口只保留卡片右上角删除按钮，详情抽屉内不提供删除按钮。
2. 读取当前 batch。
3. 如果记录影响库存，回补生豆库存。
4. 删除 roast_batches。
5. 删除失败时回滚库存调整。

页面 mutation 仍有 Query 缓存层面的乐观移除/恢复。

21. “新增支出”只用于录入手工支出，不再用于录入生豆采购。
22. 财务手工支出台账与成本核算统一接入 PocketBase：手工支出不再依赖本地缓存持久化；页面加载、保存与刷新均直接以 PocketBase 为准。
23. 财务经营概览与成本模板之间增加利润口径说明，使用无边框的小字行内文案解释毛利润与经营利润各自的计算方式，不再单独做卡片化承载。

财务经营概览字段：

| 行标签 | 值 |
| --- | --- |
| 全部花费 | `totalExpenses` |
| 已实现收入 | `realizedIncome` |
| 当前库存预估收入 | `estimatedRevenue` |
| 毛利润 | `grossProfit` |
| 经营利润 | `operatingProfit` |

财务口径定义：

1. `全部花费` 只统计实际支出，不统计摊销口径或核算口径成本。
2. `生豆采购` 不通过手工财务台账录入，而是根据生豆记录中的 `采购日期` 与 `购买总价` 自动计入当前统计范围内的支出。
3. `包装`、`邮费` 与其他经营类支出通过“新增支出”手工录入。
4. `已实现收入` 改为按烘焙历史自动汇总：只有去向为“销售”的烘焙记录才计入，且每完成一条销售烘焙记录即按 `1 份 × 默认单份售价` 计入；去向为“自留”的烘焙记录不计入收入。
5. 旧烘焙记录默认按“销售”处理，且后续允许在烘焙历史中改成“自留”或重新改回“销售”。
6. 毛利润按“已经卖出的收入 - 生豆采购 - 包装支出 - 邮费支出”理解。
7. 经营利润按“已经卖出的收入 - 当前时间范围内全部实际支出”理解。
8. `当前库存预估收入` 继续根据剩余生豆、默认售价、默认零售规格和最近脱水率估算。

新增支出抽屉：

1. 字段只保留 `日期 / 类别 / 金额 / 备注`。
2. 类别顺序为 `包装 / 邮费 / 折旧 / 其他 / 自定义`，其中“自定义”固定放在最底部，并允许创建和复用自定义类别。
3. 自定义类别通过历史支出记录沉淀，后续新增支出时继续可选。
4. 备注字段默认不再展示“可留空”辅助文案，仅在校验失败时显示错误提示。
5. 移动端新增支出表单采用扁平化 grouped list 视觉：基础字段为第一组，备注为第二组，弱化卡片感，保持黑白灰与浅分隔线层级。

财务台账同步补充：

1. 手工支出记录远端集合为 `finance_expense_records`。
2. 财务页与全局快速刷新会同时同步 `成本核算 + 财务台账`，并把远端最新记录回填到内存态与 Query Cache。
3. 当 PocketBase 未配置、网络不可用或远端集合尚未建立时，手工支出会直接报错提示，不再降级写入本地持久化缓存。

烘焙历史补充：

1. 烘焙记录新增 `去向` 字段，选项为 `销售 / 自留`。
2. 新建烘焙记录时默认 `销售`。
3. 去向可在烘焙记录完整编辑与字段级编辑中后续修改。
4. 去向展示与保存以 PocketBase 返回结果为准，不依赖本地 bootstrapped 初值或前端乐观缓存先行覆盖。
5. 去向为 `销售` 时，熟豆镜像价格使用生豆中配置的 `默认单份售价`，并参与财务页 `已实现收入` 汇总。
6. 去向为 `自留` 时，熟豆镜像价格按 `（生豆价格 / 1000）× 默认烘焙量` 计算，不参与财务页 `已实现收入` 汇总。

生豆采购补充：

1. 生豆创建和编辑时必须维护 `采购日期`。
2. 生豆采购日期作为财务统计中生豆采购支出的归属日期。
3. 当生豆采购日期或购买总价被修改时，财务统计结果会自动同步变化。
4. 删除生豆后，对应生豆采购支出也会随之从统计结果中消失。

### 12.2 Repository 选择策略

生豆、烘焙计划、烘焙记录服务层都遵循类似策略：

```text
test 环境
  -> mock/local repository
未配置连接
  -> mock/local repository
已配置连接
  -> REST repository
```

财务模块当前拆分为两类持久化路径：

```text
成本核算
  -> PocketBase 远端优先
  -> 当前运行期内存态回填
  -> 快速刷新时重新拉取远端

手工收入 / 手工支出
  -> PocketBase 远端优先
  -> 当前运行期内存态回填
  -> 快速刷新时重新拉取远端
```

注意：主业务仓储默认使用 PocketBase 配置与 `PocketBaseRestClient`；只有熟豆镜像与设置页熟豆探活在 `roastedBean` 为 Supabase URL 时使用 `RoastedBeanSupabaseDataClient`。

### 12.3 乐观更新模式

常见写入模式：

```text
onMutate
  -> cancelQueries
  -> 保存 previous cache
  -> 写入 optimistic cache
mutationFn
  -> 调用 service / repository
onError
  -> 恢复 previous cache
  -> 恢复本地 optimistic snapshot
onSuccess
  -> 用服务端返回值替换缓存
```

页面层额外使用：

```text
submissionBackupService.save(action, payload, domain)
```

用于在失败或异步同步前保留提交内容。

### 12.4 本地待同步

生豆新增特殊分支：

| 错误 | 行为 |
| --- | --- |
| NETWORK | 乐观记录保存为 pending，本地待同步 |
| HTTP 404 | 视为远端暂未就绪，本地待同步 |
| 其他错误 | 回滚 optimistic bean |

快速刷新时：

```text
beanSyncService.getPendingOperations()
  -> beanService.syncPendingOperations()
```

会尝试把待同步操作上传。

补充约束：

```text
roastBatchService.syncLocalAndRemote()
roastPlanService.syncLocalAndRemote()
  -> 只同步真实远端记录
  -> 跳过 id 以 local- 开头的乐观临时记录
  -> 刷新完成后继续保留这些乐观记录，等待原始创建请求完成
```

### 12.5 熟豆镜像

启用条件：

```text
roastedBean connection 已配置
非 test 环境
```

镜像目标：

```text
coffee_beans
```

镜像数据包含：

```text
烘焙记录字段
生豆来源字段
默认售价/规格
拼配组件
熟豆展示字段
状态、图片、评分预留字段
```

名称规则：

```text
优先使用：熟豆名称 + 空格 + 处理法
若熟豆名称为空：生豆名称 + 空格 + 处理法
若处理法为空：仅保留可用名称本身
```

烘焙商规则：

```text
优先读取当前登录用户昵称
即 PocketBase users.name
若昵称为空，则 roaster 保持空字符串
```

风味规则：

```text
生豆主档在 PocketBase 里使用 flavor_tags 单字段存储
格式为逗号分隔字符串，例如：柑橘,花香,莓果
前端读取后解析成 string[] 标签数组
创建烘焙历史同步到熟豆 Supabase 时，写入 flavor: string[]
例如：["柑橘","花香","莓果"]
```

烘焙后处理规则：

```text
生豆主档使用 aging_days / tasting_end_days 存储
创建烘焙历史同步到熟豆 Supabase 时：
startDay = agingDays
endDay = tastingEndDays
默认值分别为 14 / 40
```

用途：把 EasyBake 生产出的熟豆记录同步到熟豆展示或销售系统。

## 17. 当前实现的维护注意点

1. 熟豆 Supabase 专属能力集中在 `RoastedBeanSupabaseDataClient`、`roastedBeanSupabaseConnectionSyncService` 与 `roastedCoffeeBeanMirrorService`；除这两条链路外，主业务数据默认按 PocketBase 理解。
2. `pocketBaseSessionService` 是内存会话，不是持久 localStorage token。若后续要刷新后保持登录，需要优先通过 HttpOnly Cookie + BFF 方案，而不是把 token 放入 localStorage。
3. Settings 的连接配置服务目前也是内存状态服务，远端同步通过 app settings 补齐。若做离线持久化，需要谨慎设计迁移和清理策略。
4. 烘焙记录创建、更新、删除会影响生豆库存，是高风险交互；调整前必须补充库存回滚测试。
5. 熟豆镜像失败时有回滚主记录逻辑；如果未来改成最终一致性，需要明确用户可见状态和补偿任务。
6. Finance 已落独立交互页面，但真实销售订单与财务自动汇总仍未接入；当前收入、经营费用属于前端本地台账，后续扩展时应继续保持领域拆分，不要把业务逻辑塞回现有页面。
7. 页面文件中已有一些中文注释和业务流程注释，新增代码仍应保持组件单一职责，避免页面继续膨胀。
