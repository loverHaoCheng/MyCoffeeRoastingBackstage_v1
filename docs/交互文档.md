> 本文档生成日期：2026-07-08  
> 适用范围：当前代码库 `src/`、`server/`、`supabase/` 与既有文档中已经落地的交互。

## 1. 项目定位

EasyBake 是一个 React + TypeScript 的咖啡烘焙管理后台，目前核心围绕单用户工具形态展开，已具备向多用户、权限、工作流、插件系统扩展的基础。

当前交互覆盖以下业务域：

| 业务域 | 当前入口 | 主要能力 | 当前状态 |
| --- | --- | --- | --- |
| Auth | `#/login`、`#/register` | PocketBase 账号登录、注册、恢复会话、退出登录 | 已接入 |
| Bean | `#/beans` | 生豆库存、创建、搜索、查看、字段级编辑、删除、待同步本地缓存 | 已接入 |
| Roast | `#/roasts` | 烘焙计划列表、手动创建、JSON 导入、查看、编辑、删除 | 已接入 |
| Production | `#/production` | 烘焙历史、创建烘焙记录、库存扣减、字段级编辑、删除 | 已接入 |
| Settings | `#/settings` | 熟豆库连接、界面外观、卡片展示、成本模板、二维码、版本号 | 已接入 |
| Finance | 无独立页面入口 | 成本计算同步能力被 `appDataRefresh` 引用 | 后续扩展 |
| Inventory | 无独立页面入口 | 目录预留 | 后续扩展 |

## 2. 技术与运行框架

| 层级 | 实现 | 交互意义 |
| --- | --- | --- |
| 应用框架 | React 19 + TypeScript 5 + Vite | 单页 HashRouter 应用，适合静态部署与后台管理 |
| UI 框架 | Ant Design 5 | 表单、按钮、抽屉、菜单、提示、弹窗、标签等基础交互 |
| 路由 | React Router `createHashRouter` | 使用 `#/path` 路由，刷新页面不依赖服务端路由转发 |
| 数据请求 | TanStack Query | 列表查询、缓存、乐观更新、失败回滚 |
| 全局状态 | Zustand | 鉴权状态、设置状态、布局状态 |
| 表单校验 | AntD Form、React Hook Form、Zod | 登录注册使用 AntD Form；业务表单使用 schema 校验 |
| 数据源 | PocketBase REST 命名 + Supabase REST 兼容层 | 代码中保留 `Supabase*` 历史命名，但多个客户端通过 `PocketBaseRestClient` 复用 REST 能力 |
| 测试 | Vitest + React Testing Library | 已覆盖路由、页面、同步、服务、设置等关键交互 |

## 3. 应用初始化交互

### 3.1 启动顺序

入口链路：

```text
main.tsx
  -> App
    -> AppProviders
    -> AppAuthBootstrap
    -> AppUpdateBanner
    -> AppRealtimeSync
    -> RouterProvider
```

启动时会发生：

1. 清理废弃 localStorage key。
2. 空闲时预加载 auth、bean、production、roast、settings 模块。
3. 初始化 Ant Design 中文配置、黑白灰主题 token、QueryClient、ErrorBoundary。
4. `AppAuthBootstrap` 调用 `useAuthStore.bootstrapSession()` 恢复登录态。
5. `AppRealtimeSync` 在登录后建立 PocketBase realtime 订阅并触发数据刷新。

### 3.2 Provider 交互

`AppProviders` 统一提供：

| 能力 | 行为 |
| --- | --- |
| QueryClient | 默认 `staleTime: Infinity`，避免频繁自动刷新；由手动刷新、实时事件、业务 mutation 主动更新 |
| AntD ConfigProvider | 统一中文、主题、圆角、字体、主色黑白灰 |
| AntD App | 提供 `message`、`modal` 等反馈 API |
| ErrorBoundary | 捕获渲染异常，避免整页崩溃 |
| 主题同步 | 根据设置页 `themeMode` 写入 `document.documentElement.dataset.appTheme` |
| 消息定位 | 根据应用头部高度更新 AntD message top，避免遮挡顶部导航 |

## 4. 路由与鉴权

### 4.1 路由表

| 路由 | 页面 | 访问条件 | 备注 |
| --- | --- | --- | --- |
| `#/` | 重定向到 `#/beans` | 已登录 | 默认首页 |
| `#/beans` | `BeanPage` | 已登录 | 生豆库存 |
| `#/roasts` | `RoastPage` | 已登录 | 烘焙计划 |
| `#/production` | `ProductionPage` | 已登录 | 烘焙历史 |
| `#/settings` | `SettingsPage` | 已登录 | 设置 |
| `#/login` | `LoginPage` | 未登录 | 已登录则跳到 `#/beans` |
| `#/register` | `RegisterPage` | 未登录 | 已登录则跳到 `#/beans` |
| `*` | 重定向到 `#/` | 任意 | 兜底 |

### 4.2 鉴权守卫

`RequireAuth`：

1. `hasHydrated === false` 或 `status === 'hydrating'` 时显示全屏 Spin：`正在恢复登录态`。
2. 非 authenticated 时跳转 `#/login`。
3. 已登录时渲染 `MainLayout`。

`PublicOnly`：

1. 恢复登录态期间显示同一 loading。
2. 已登录访问登录/注册页时重定向 `#/beans`。
3. 未登录时渲染登录或注册页。

### 4.3 登录注册交互

登录页字段：

| 字段 | 校验 | 行为 |
| --- | --- | --- |
| 邮箱 | 必填、邮箱格式 | 提交给 `/api/auth/login` |
| 密码 | 必填 | 提交给 `/api/auth/login` |

注册页字段：

| 字段 | 校验 | 行为 |
| --- | --- | --- |
| 邮箱 | 必填、邮箱格式 | 提交给 `/api/auth/register` |
| 密码 | 必填、至少 8 位 | 注册成功后自动登录 |
| 确认密码 | 必填、必须等于密码 | 不一致时阻止提交 |

成功后：

1. 清理本地应用状态。
2. 保存内存会话。
3. 更新 Zustand 鉴权状态。
4. 提示成功。
5. 跳转 `#/beans`。

退出登录：

1. 用户在设置页顶部账号栏点击 `退出登录`。
2. 请求 `/api/auth/logout`，失败时记录 warn 但仍清理本地会话。
3. 清空 QueryClient 缓存。
4. 跳转 `#/login`。

维护注意：`pocketBaseSessionService` 当前是内存会话，虽然暴露 storage key，但 `load/save` 未写入浏览器存储。会话恢复依赖 `/api/auth/session` 与当前运行环境。

## 6. 全局刷新与实时同步

### 6.1 快速刷新

触发入口：

| 入口 | 触发条件 |
| --- | --- |
| 桌面/非触控悬浮刷新按钮 | 用户点击 |
| 触控端下拉刷新 | 用户下拉 |
| Realtime 事件 | 订阅主题发生变化 |
| 页面可见性恢复 | 页面从隐藏变为可见 |
| 网络恢复 | `online` 事件 |

刷新范围按当前路由决定：

| 当前路由 | 刷新范围 |
| --- | --- |
| `/beans` | 生豆 |
| `/roasts` | 生豆、烘焙计划、烘焙记录 |
| `/production` | 生豆、烘焙记录 |
| `/settings` | 设置相关 |
| all | 生豆、烘焙计划、烘焙记录、成本核算 |

刷新主流程：

```text
refreshQuickAppData
  -> 清理废弃本地 key
  -> 载入连接配置、成本模板、显示设置
  -> 同步生豆待处理操作
  -> 按当前 scope 同步远端/本地数据
  -> 回填 TanStack Query 缓存
  -> 返回 success / failed / downloaded / uploaded
```

### 6.2 Realtime 订阅

登录且页面可见时创建 `EventSource`：

```text
/api/realtime
```

订阅主题：

| 主题 | 影响页面 |
| --- | --- |
| `app_settings/*` | 生豆、烘焙历史、烘焙计划、设置 |
| `bean_sale_specs/*` | 生豆 |
| `green_beans/*` | 生豆、烘焙历史、烘焙计划 |
| `green_bean_purchase_batches/*` | 生豆 |
| `roast_batches/*` | 生豆、烘焙历史、烘焙计划 |
| `roast_profiles/*` | 烘焙历史、烘焙计划 |

事件刷新有 320ms debounce，避免高频事件导致连续请求。

`BeanPage` 结构：

```text
搜索栏
库存概览
  - 总剩余库存
  - 均价
生豆列表
  - 有库存卡片
  - 零库存折叠区
新增生豆 FAB
新增生豆抽屉
详情抽屉
字段编辑抽屉
```

### 8.2 搜索与列表

搜索字段匹配：

```text
bean.name
bean.origin
bean.process
bean.grade
```

列表分组：

| 分组 | 条件 | 默认展示 |
| --- | --- | --- |
| 有库存 | `stockKg > 0` | 直接展示 |
| 零库存 | `stockKg === 0` | 默认折叠 |

空状态：

```text
没有匹配的生豆批次
```

加载状态：

1. 首次请求且本地无缓存时显示 Spin。
2. 有缓存时继续展示缓存，后台请求更新。

### 8.3 生豆卡片

卡片标题：

```text
bean.name
```

副标题：

```text
编号 {bean.code}
或
编号 {bean.id}
```

可展示字段包括：

```text
库存、成本、成本模板、供应商、处理法、国家、产区、小产区、处理厂、品种、等级、产季、编号、默认烘焙量、默认单份售价、默认单份重量、采购总重、采购总价、剩余重量、海拔、含水率、密度、备注
```

预览字段由设置页 `beanInventory` 卡片展示配置决定。`local-` 开头的本地草稿会展示所有字段，帮助用户识别待同步数据。

卡片头部交互：

1. 名称与编号上下分两行显示。
2. 右侧操作按钮顺序为查看、全部编辑、删除。
3. 点击“全部编辑”直接打开完整生豆表单抽屉。
4. 点击字段行仍保留单字段快捷编辑能力。

### 8.4 新增生豆

点击 FAB 前置检查：

1. 生豆数据库连接必须存在。
2. 至少有一个成本模板。

不满足时提示：

| 条件 | 提示 |
| --- | --- |
| 未配置生豆连接 | 请先前往设置页创建并连接生豆数据库，完成后才能新增数据。 |
| 无成本模板 | 请先前往设置页创建至少一个成本模板，再新增生豆。 |

新增抽屉包含两个 Tab：

| Tab | 状态 |
| --- | --- |
| 界面创建 | 已实现，使用 `BeanManualCreator` + `BeanForm` |
| AI 图片识别 | 占位展示，后续扩展 |

默认表单值：

```text
默认烘焙量 200g
采购总重 1000g
剩余重量 1000g
其他文本字段为空
成本模板默认自动带入
```

提交链路：

```text
BeanManualCreator
  -> BeanForm 校验
  -> BeanPage.handleCreateBean
  -> submissionBackupService.save(create)
  -> beanService.createOptimisticBean
  -> queryClient.setQueryData 插入乐观记录
  -> beanService.createRemoteBean
    -> 成功：finalizeOptimisticBean，替换为远端记录
    -> 网络/404：persistOptimisticBeanAsPending，本地待同步
    -> 其他错误：rollbackOptimisticBean，回滚并提示
```

### 8.5 查看详情

点击卡片眼睛图标：

1. 设置 `selectedBeanId`。
2. 清空 `selectedBeanFieldPath`。
3. `detailMode = 'view'`。
4. 打开详情抽屉。

详情展示：

```text
名称、产地、处理法、等级、总库存、剩余库存、成本、成本模板、默认烘焙量、默认单份售价、默认单份重量、供应商、更新时间、豆种、产季、编号
```

### 8.6 字段级编辑

点击卡片字段行：

1. 根据字段 key 映射到 `GreenBeanFormInput` fieldPath。
2. 打开 `BeanFieldEditorDrawer`。
3. 抽屉先用卡片数据生成 fallback draft。
4. 同时请求 `beanService.getEditableBean` 获取完整可编辑详情。
5. 用户已编辑时不会被异步详情覆盖。

字段控件类型：

| 字段类型 | 控件 |
| --- | --- |
| 重量、价格、海拔、密度、含水率 | `InputNumber` |
| 成本模板 | `Select` |
| 备注 | `Input.TextArea` |
| 文本字段 | `Input` |

提交链路：

```text
Zod schema 校验
  -> 失败：提示第一条错误并滚动到字段
  -> 成功：关闭抽屉
      -> submissionBackupService.save(update)
      -> useUpdateBean.mutateAsync
      -> onMutate 乐观更新列表和详情缓存
      -> onError 回滚
      -> onSuccess 写入远端返回值
```

### 8.7 删除生豆

点击删除：

1. `modal.confirm` 二次确认。
2. 确认文案提示会一并删除关联采购批次、烘焙记录等数据。
3. `useDeleteBean` 先从 Query 缓存移除。
4. `beanService.prepareOptimisticDelete` 保存删除快照。
5. 成功后清理详情缓存并 invalidate 列表。
6. 失败时恢复列表、详情与本地快照。

```text

```

### 9.5 JSON 导入

JSON 导入提供：

| 控件 | 行为 |
| --- | --- |

```text

  -> 插入 Query 缓存
  -> roastBatchService.createBatch
```

`roastBatchService.createBatch` 内部副作用：

```text

  -> beanService.adjustRemainingWeight(greenBeanId, inputWeightGrams)
  -> 创建 roast_batches 记录
  -> 如果熟豆库镜像启用
      -> roastedCoffeeBeanMirrorService.syncCreatedBatch
      -> 镜像失败时尝试删除主库 roast batch
  -> 保存本地批次缓存
```

失败回滚：

| 失败位置 | 回滚策略 |
| --- | --- |
| 创建主记录失败 | 回补生豆库存 |
| 库存回滚也失败 | 抛出需要人工核对的 AppError |
| 熟豆镜像失败 | 尝试删除主记录；删除失败则抛出主库/熟豆库不一致风险 |
| 页面层捕获失败 | 回滚 optimistic batch 并提示 |

### 10.5 更新与删除记录

更新记录时：

1. 卡片右上角“全部编辑”或详情内编辑入口会打开完整 `RoastBatchDrawer mode="edit"`。
2. 完整编辑抽屉底部统一为“取消 / 保存烘焙记录”，并使用保存图标；按钮组位于内容滚动区内，复用 `DrawerActionBar compact` 的 sticky 样式。
3. 单字段编辑继续使用 `RoastBatchFieldEditorDrawer`，底部统一为“取消 / 保存字段名”。
4. 提交更新前先读取当前 batch。
5. 计算当前库存影响重量与下一状态库存影响重量。
6. 同一生豆只调整差值。
7. 更换生豆时先回补旧生豆，再扣减新生豆。
8. 更新失败时按反向 rollbacks 回滚库存。

删除记录时：

1. 读取当前 batch。
2. 如果记录影响库存，回补生豆库存。
3. 删除 roast_batches。
4. 删除失败时回滚库存调整。

页面 mutation 仍有 Query 缓存层面的乐观移除/恢复。

## 17. 当前实现的维护注意点

1. `PocketBaseRestClient` 当前从 `supabaseRestClient` re-export，代码命名有历史包袱。维护时应以实际客户端行为为准，不要仅根据类名判断数据源。
2. `pocketBaseSessionService` 是内存会话，不是持久 localStorage token。若后续要刷新后保持登录，需要优先通过 HttpOnly Cookie + BFF 方案，而不是把 token 放入 localStorage。
3. Settings 的连接配置服务目前也是内存状态服务，远端同步通过 app settings 补齐。若做离线持久化，需要谨慎设计迁移和清理策略。
4. 烘焙记录创建、更新、删除会影响生豆库存，是高风险交互；调整前必须补充库存回滚测试。
5. 熟豆镜像失败时有回滚主记录逻辑；如果未来改成最终一致性，需要明确用户可见状态和补偿任务。
6. Finance 与 Inventory 尚未有独立交互页面，新增时应按 Feature First 建模块，不要把业务逻辑塞入现有页面。
7. 页面文件中已有一些中文注释和业务流程注释，新增代码仍应保持组件单一职责，避免页面继续膨胀。
