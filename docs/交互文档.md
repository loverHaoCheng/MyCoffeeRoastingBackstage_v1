## 1. 项目定位

EasyBake 是一个 React + TypeScript 的咖啡烘焙管理后台，目前核心围绕单用户工具形态展开，已具备向多用户、权限、工作流、插件系统扩展的基础。

当前交互覆盖以下业务域：

| 业务域 | 当前入口 | 主要能力 | 当前状态 |
| --- | --- | --- | --- |
| Auth | `#/login`、`#/register` | PocketBase 账号登录、注册、恢复会话、退出登录 | 已接入 |
| Legal | `#/terms`、`#/privacy`、`#/data-deletion` | 用户协议、隐私政策、数据删除机制公开展示 | 已接入 |
| Bean | `#/beans` | 生豆库存、创建、搜索、查看、字段级编辑、删除、待同步本地缓存 | 已接入 |
| Roast | `#/roasts` | 烘焙计划列表、手动创建、JSON 导入、查看、编辑、删除 | 已接入 |
| Production | `#/production` | 烘焙历史、创建烘焙记录、库存扣减、字段级编辑、删除 | 已接入 |
| Settings | `#/settings` | 熟豆库连接、界面外观、卡片展示、二维码、版本号 | 已接入 |
| Finance | `#/finance` | 经营总览、支出台账、成本模板 | 已接入 |
| Inventory | 无独立页面入口 | 目录预留 | 后续扩展 |

## 2. 技术与运行框架

| 层级 | 实现 | 交互意义 |
| --- | --- | --- |
| 应用框架 | React 19 + TypeScript 5 + Vite | 单页 HashRouter 应用，适合静态部署与后台管理 |
| UI 框架 | Ant Design 5 | 表单、按钮、抽屉、菜单、提示、弹窗、标签等基础交互 |
| 路由 | React Router `createHashRouter` | 使用 `#/path` 路由，刷新页面不依赖服务端路由转发 |
| 数据请求 | TanStack Query | 列表查询、缓存、乐观更新、失败回滚 |
| 全局状态 | Zustand | 鉴权状态、设置状态、布局状态 |
| 表单校验 | AntD Form、React Hook Form、Zod | 登录注册使用 AntD Form；业务表单使用 schema 校验；所有可编辑控件必须提供可访问名称 |
| 数据源 | PocketBase REST + 熟豆 Supabase REST | 主业务数据走 PocketBase；熟豆镜像和设置页熟豆探活单独使用 Supabase REST |
| 测试 | Vitest + React Testing Library | 已覆盖路由、页面、同步、服务、设置等关键交互 |

键盘焦点：所有可聚焦控件在 `:focus-visible` 状态下必须显示高对比度黑白灰焦点环；鼠标点击不应触发该焦点环。

## 3. 应用初始化交互

### 3.1 启动顺序

入口链路：

```text
main.tsx
  -> App
    -> AppProviders
    -> AppAuthBootstrap
    -> AppUpdateBanner
    -> AppRealtimeSync
    -> RouterProvider
```

启动时会发生：

1. 清理废弃 localStorage key。
2. 应用壳、认证页与业务页面均由路由按需加载；启动阶段不得预加载未访问的库存、烘焙、生产、财务与设置模块。
3. 初始化 Ant Design 中文配置、黑白灰主题 token、QueryClient、ErrorBoundary。
4. `AppAuthBootstrap` 调用 `useAuthStore.bootstrapSession()` 恢复登录态。
5. `AppRealtimeSync` 在登录后建立 PocketBase realtime 订阅并触发数据刷新。

### 3.2 Provider 交互

`AppProviders` 统一提供：

| 能力 | 行为 |
| --- | --- |
| QueryClient | 默认 `staleTime: Infinity`，避免频繁自动刷新；由手动刷新、实时事件、业务 mutation 主动更新 |
| AntD ConfigProvider | 统一中文、主题、圆角、字体、主色黑白灰 |
| AntD App | 提供 `message`、`modal` 等反馈 API |
| ErrorBoundary | 捕获渲染异常，避免整页崩溃 |
| 主题同步 | 根据设置页 `themeMode` 写入 `document.documentElement.dataset.appTheme` |
| 消息定位 | 根据应用头部高度更新 AntD message top，避免遮挡顶部导航 |

### 4.2 鉴权守卫

`RequireAuth`：

1. `hasHydrated === false` 或 `status === 'hydrating'` 时显示全屏 Spin：`正在恢复登录态`。
2. 非 authenticated 时跳转 `#/login`。
3. 已登录时渲染 `MainLayout`。

`PublicOnly`：

1. 恢复登录态期间显示同一 loading。
2. 已登录访问登录/注册页时重定向 `#/beans`。
3. 未登录时渲染登录或注册页。

### 4.3 登录注册交互

登录页字段：

| 字段 | 校验 | 行为 |
| --- | --- | --- |
| 邮箱 | 必填、邮箱格式 | 提交给 `/api/auth/login`；也作为重发验证邮件的目标邮箱 |
| 密码 | 必填 | 提交给 `/api/auth/login` |
| 协议确认 | 必须勾选 | 用户需阅读并同意《用户协议》和《隐私政策》后才能提交登录 |

认证页布局约束：

1. 登录页与注册页的表单块使用同一组纵向定位规则。
2. 默认保持在屏幕中线略靠上位置，避免在移动端落到视口下半区。
3. 无论表单内容后续如何增长，顶部都至少保留 `10vh` 间距。
4. 登录页、注册页、找回密码页顶部统一显示 `EasyBake` 品牌标题。
5. 找回密码页不再展示独立的 `PocketBase Auth / 找回密码 / 描述` 头部文案。
6. 登录页、注册页、找回密码页底部通过统一 `LegalFooter` 展示 ICP 备案号，备案信息跟随页面自然滚动，不使用固定悬浮底栏。

注册页字段：

| 字段 | 校验 | 行为 |
| --- | --- | --- |
| 邮箱 | 必填、邮箱格式 | 提交给 `/api/auth/register` |
| 密码 | 必填、至少 8 位 | 创建账号后等待邮箱验证 |
| 确认密码 | 必填、必须等于密码 | 不一致时阻止提交 |
| 协议确认 | 必须勾选 | 用户需阅读并同意《用户协议》和《隐私政策》后才能提交注册 |

注册成功后：

1. 清理本地应用状态。
2. 调用 `/api/auth/register` 创建账号；BFF 仅在该首次创建请求中将注册邮箱写入 `users.name` 作为默认昵称，后续邮箱变化不会覆盖该字段，用户可自行修改昵称。
3. BFF 触发 PocketBase `request-verification`。
4. 不写入登录态，不更新 authenticated 状态。
5. 跳转回 `#/login`，提示用户先完成邮箱验证。

合规公开页面：

1. `#/terms` 展示用户协议。
2. `#/privacy` 展示隐私政策。
3. `#/data-deletion` 展示数据删除机制。
4. 三个页面未登录可访问，顶部提供互相跳转，并展示生效日期、运营主体和联系邮箱。
5. 当前运营主体为个人开发者，合规联系邮箱为 `zhc1501705072@163.com`。
6. 法律页面使用独立纵向滚动容器，避免受全局 `body` 固定视口影响导致长文本无法下滑。
7. 法律页面文章底部提供 `退出` 按钮，点击后回到 `#/login`。
8. 法律页面和应用根容器禁用横向滚动与浏览器横向 overscroll，内容滚动只保留纵向。
9. 法律页面最底部复用 `LegalFooter` 展示 ICP 备案号；公安联网备案完成后在 ICP 备案号下方追加公安备案号。

设置页账号注销交互：

1. 设置页底部新增 `注销账号` 危险按钮，不再额外展示说明卡片；`当前 Web 上传版本` 展示在该按钮下方。
2. 账号注销按钮默认开放，用户点击 `注销账号` 会弹出确认框。
3. 确认框会明确提示账号、烘焙业务数据、财务记录、库存设置和关联配置会被永久删除且无法恢复。
4. 确认框需要等待 5 秒倒计时结束后，`确认注销` 才可点击。
5. 确认后调用 `/api/auth/account`，删除当前 PocketBase 用户账号及其全部关联业务数据。
6. 删除成功后清空本地登录态、设置缓存、查询缓存，以及当前站点的 `localStorage`、`sessionStorage`、可访问 Cookie、IndexedDB 与 Cache Storage，并回到未登录状态；`HttpOnly` 认证 Cookie 由服务端响应过期。
7. 设置页底部在版本号下方展示 ICP 备案号，作为小字号合规信息，不影响设置项布局和危险操作按钮。

登录限制：

1. `/api/auth/login` 成功鉴权后，还会检查 `record.verified`。
2. 若邮箱未验证，则拒绝写入 Cookie，并提示先完成验证。
3. `/api/auth/session` 刷新登录态时也会执行同样的验证检查，避免旧版本自动登录的未验证账号继续进入系统。

辅助认证操作：

1. 登录页支持“重新发送验证邮件”，调用 `/api/auth/request-verification`。
2. `#/forgot-password` 支持发送密码重置邮件，调用 `/api/auth/request-password-reset`。
3. 两个接口对邮箱不存在等客户端错误统一返回成功文案，对上游 5xx 仍向前端抛错，便于发现 SMTP 或 PocketBase 异常。

退出登录：

1. 用户在设置页顶部账号栏点击 `退出登录`。
2. 请求 `/api/auth/logout`，失败时记录 warn 但仍清理本地会话。
3. 清空 QueryClient 缓存，以及当前站点的 `localStorage`、`sessionStorage`、可访问 Cookie、IndexedDB 与 Cache Storage；`HttpOnly` 认证 Cookie 由服务端响应过期。
4. 跳转 `#/login`。

维护注意：`pocketBaseSessionService` 仅保存当前运行期的公开用户资料，`load/save` 不写入浏览器存储，也不保存 PocketBase Token。会话恢复依赖 `/api/auth/session` 与当前运行环境。

移动端交互特性：

| 特性 | 行为 |
| --- | --- |
| 底部导航 | 仅保留生豆库存 / 烘焙计划 / 烘焙历史 / 财务四个业务入口，并根据当前路由移动 active pill |
| 顶部右侧动作 | 当前页面如注册新增动作，则在 Header 右侧显示对应加号入口；设置面板展开时隐藏 |

| 触控下拉刷新 | 粗指针设备启用 `GlobalPullToRefresh`；明显横向滑动会优先锁定为横滑手势，不进入下拉刷新状态 |
| 主内容区横滑导航 | 仅在主内容区域生效；左向右滑动切换到左侧一级菜单，位于最左侧菜单时打开设置抽屉；设置抽屉已打开时继续左向右滑不再触发；设置抽屉已打开时右向左滑优先关闭抽屉，普通页面右向左滑切换到右侧一级菜单 |

主内容区横滑导航避让规则：

1. Header、底部导航不作为触发区域；设置抽屉内部仅支持右向左滑关闭设置，左向右滑不触发。
2. 表单输入、按钮、链接、弹窗、抽屉、选择器、Tabs、表格、横向滚动区域不触发全局横滑导航。
3. 复杂组件如图表、横向列表、可拖拽区域可以添加 `data-prevent-swipe-navigation="true"` 主动退出全局横滑接管。
4. 手势必须满足明显横向位移后才触发，避免与纵向滚动、下拉刷新冲突。
5. 当触摸轨迹被判定为横向手势后，下拉刷新在本次触摸周期内保持禁用，避免左滑或右滑时误触发刷新提示。

### 6.2 Realtime 订阅

登录且页面可见时创建 `EventSource`：

```text
/api/realtime
```

浏览器不持有 PocketBase Token。实时 SSE 连接和后续订阅请求都只携带同源 `HttpOnly Cookie`；BFF 在服务端校验 Cookie 并向 PocketBase 注入认证信息。业务集合请求同样只允许经 BFF 的白名单路由访问。

订阅主题：

| 主题 | 影响页面 |
| --- | --- |
| `app_settings/*` | 生豆、烘焙历史、烘焙计划、设置 |
| `bean_sale_specs/*` | 生豆 |
| `green_beans/*` | 生豆、烘焙历史、烘焙计划 |
| `green_bean_purchase_batches/*` | 生豆 |
| `roast_batches/*` | 生豆、烘焙历史、烘焙计划 |
| `roast_profiles/*` | 烘焙历史、烘焙计划 |

事件刷新有 320ms debounce，避免高频事件导致连续请求。

### 6.3 主库认证边界与 BFF 代理

主库请求链路：

```text
Browser
  -> 同源 /api/* 请求（credentials: same-origin）
  -> Nginx
  -> Node BFF
  -> 从 HttpOnly Cookie 读取 PocketBase Token
  -> 127.0.0.1:8090 PocketBase
```

认证交互规则：

1. `/api/auth/login`、`/api/auth/session` 与 `/api/auth/profile` 成功时只返回 `{ record }`；响应不得包含 `token`。
2. BFF 将 PocketBase Token 写入 `HttpOnly; SameSite=Lax; Secure` Cookie。浏览器 JavaScript 不得读取、持久化或在 `Authorization` 请求头中转发该 Token。
3. 前端运行期会话仅保存用户 `id`、`email`、昵称、验证状态等公开资料；刷新页面后必须通过 `/api/auth/session` 恢复。
4. 主业务 `PocketBaseRestClient` 固定访问当前浏览器同源 `/api/collections/*`，即使界面中存在历史主库地址配置，也不得绕过 BFF 直连外部 PocketBase。
5. 外部熟豆镜像是例外：`roastedBean` 配置为 Supabase URL 时，继续使用专用 Supabase 客户端；配置为独立 PocketBase 时需显式关闭主库 BFF 传输，不能复用主库认证 Cookie。

BFF 仅允许以下主业务集合通过浏览器代理访问：

```text
app_settings
bean_sale_specs
cost_calculations
finance_expense_records
green_beans
green_bean_purchase_batches
roast_batches
roast_profiles
roast_records
```

| 模块 | 职责 |
| --- | --- |
| `router.ts` | 只负责 `/api/*` 路由分发与方法限制 |
| `config.ts`、`types.ts`、`utils.ts` | 环境变量、共享类型和基础类型守卫 |
| `http.ts` | JSON 请求/响应、Cookie、通用 API 成功/失败响应 |
| `pocketbase-client.ts`、`record-utils.ts` | PocketBase URL、响应归一化、白名单代理、记录列表与删除工具 |
| `auth-common.ts`、`auth-handlers.ts` | 登录、注册、会话刷新、昵称更新、验证邮件、找回密码、退出登录 |

| `account-handler.ts` | 账号注销及关联业务数据清理 |

BFF 入口 `server/pocketbase-auth-bff.ts` 只允许保留 handler 组装、异常兜底、独立 HTTP 服务启动逻辑；不得再把业务处理器堆回入口文件。

### 6.4 BFF 与 Nginx 发布约束

生产部署必须满足：

1. PocketBase 监听服务器内网地址 `127.0.0.1:8090`；BFF 的 `PB_BASE_URL` 默认值和生产环境显式配置都必须指向该内网地址。前端开发代理的 `VITE_DEV_API_PROXY_TARGET` 是浏览器到 BFF 的入口地址，不能复制为 `PB_BASE_URL`，否则会形成 BFF -> Nginx -> BFF 的代理循环。
2. 一个 systemd 服务的多个 drop-in 不得重复设置 `PB_BASE_URL`；后加载的配置会覆盖前项。更新前应使用 `systemctl` 或 `grep` 确认最终只有一个该变量定义。
3. Nginx 的 `/api/*` 必须整体转发至 `127.0.0.1:3001`，并启用 `proxy_buffering off` 与 `proxy_read_timeout 360s`，以支持 realtime SSE；不得保留将 `/api/collections/*` 直连 `8090` 的 location。
4. BFF 构建产物和前端构建产物需要分别发布。`./deploy.sh` 会先将前端构建结果复制到本机临时快照，再构建 BFF；BFF 候选目录通过远端 `dist/server.next` 路径部署、重启并完成认证探测后，才同步前端快照，避免将 `dist/server` 发布到静态站点目录。
5. BFF 更新时，脚本会保留当前 `dist/server` 目录为 `dist/server.previous`。若服务重启或本机 `POST /api/auth/login` 探测不是 `400`，自动恢复 `.previous` 目录并尝试重启旧服务；BFF 验证失败时不得发布前端快照。
6. 前端临时快照目录默认由 `mktemp` 创建，权限可能为 `700`。同步完成后，脚本必须将远端静态站点根目录设为至少 `755`，保证 Nginx 工作进程可穿透目录读取 `index.html` 与 `version.json`。
7. 前端发布采用版本目录：构建产物先上传至 `/var/www/easybake-releases/<version>.next`，确认含有 `index.html` 和 `version.json` 后提升为正式版本目录；`/var/www/easybake` 是 Nginx 固定入口链接，脚本以 `current.next -> current` 的原子重命名切换版本。
8. 切换前脚本把旧入口写入 `/var/www/easybake.previous`。若公网版本号、健康检查或认证探测失败，脚本必须原子切回 `previous`，而不是保留半同步或验收失败的前端版本。

发布验收：

1. `GET /api/health` 返回 `200`。
2. 未携带 Cookie 请求主业务集合返回 `401`。
3. 测试账号登录返回 `200`、设置 Cookie，并且 JSON 响应不含 `token`。
4. 携带该 Cookie 请求主业务集合返回 `200`。
5. 从服务器本机和外部网络各验证一次；若结果不同，应先检查 DNS、云网络入口或旧反向代理实例，不能将本机 curl 成功视为公网发布完成。
6. BFF 必须具备独立 HTTP 契约测试，至少覆盖健康检查、集合与实时请求的未登录拒绝、集合白名单拒绝、认证接口方法限制，以及登录成功时 JSON 不返回 Token、Token 仅写入 `HttpOnly; SameSite=Lax; Secure` Cookie。测试通过真实 Node HTTP 服务调用 BFF handler，上游 PocketBase 必须使用 mock，不能依赖生产服务。

| --- | --- |

| AI 图片识别 | 显示 `本月剩余 x / y`，其中 `y` 来自服务器配置；额度充足时打开 `AI 图片识别` 抽屉，上传图片后调用 BFF 识别并回填 `BeanForm` |
| 取消 | 关闭创建方式选择，不进入创建流程 |

AI 图片识别交互规则：

1. 用户在 `AI 图片识别` 抽屉选择 `jpeg / png / webp` 图片，原图前端限制单张不超过 20MB。
2. 前端会先把图片最长边压缩到 1600px，并导出为 2MB 以内的 JPEG Blob；data URL 仅用于本地预览。
3. 前端将压缩后的 JPEG 二进制 body 提交到 `/api/ai/bean-image-recognition`，并显示原图与压缩后的体积；BFF 内部再转换为 data URL 调用七牛云。
4. BFF 使用 HttpOnly Cookie 校验当前用户，前端不接触七牛云 API Key。
5. 创建方式选择打开时，前端先 GET `/api/ai/bean-image-recognition` 读取 `ai_usage_limits` 和当月成功日志，AI 图片识别按钮显示 `本月剩余 x / y`；额度为 0、功能关闭或额度读取失败时禁止点击，读取失败需显示服务端返回的具体原因，便于排查登录态、权限或后台额度字段配置。
6. BFF 读取 `ai_usage_limits` 判断用户是否启用与当月额度；未配置时默认 `10 次/月`。
7. BFF 调用七牛云 Qwen 时传入 `enable_thinking: false` 关闭深度思考，并通过 prompt 约束模型只输出 JSON；随后对七牛云 OpenAI-compatible 返回结构做容错解析；若模型返回可读文本但不是严格 JSON，BFF 会追加一次文本整理请求转成固定字段 JSON，整理请求同样关闭深度思考。只有识别成功、结构化字段解析成功且写入 `success` 日志后才计次，失败写 `failed` 日志但不扣次数。
8. AI 识别产季统一规范为年份后两位，例如 `2026`、`2025/26` 最终回填为 `26`。
9. 识别完成后展示字段预览和当月剩余额度。
10. 点击 `回填表单` 后在同一新增抽屉流程中切到 `新增生豆` 表单，并把识别字段合并到默认生豆表单值；生豆编号仍使用表单自动生成的 `EB-YYMMDDHHmm`，不使用图片识别出的编号覆盖。
11. 识别结果只作为待确认草稿回填到 `BeanForm`，仍需用户确认并走现有表单校验与创建链路。`BeanForm` 只协调 React Hook Form、Zod 校验、成本模板联动和提交；基础信息、烘焙后处理、产地品质、备注及操作栏分别由独立分区组件渲染，所有分区共享同一份 `control/errors`，不得建立第二套表单状态。

创建链路：

```text

  -> roastBatchService.createOptimisticBatch
  -> 插入 Query 缓存
  -> roastBatchService.createBatch
```

`roastBatchService.createBatch` 内部副作用：

```text

  -> beanService.adjustRemainingWeight(greenBeanId, inputWeightGrams)
  -> 创建 roast_batches 记录
  -> 如果熟豆库镜像启用
      -> roastedCoffeeBeanMirrorService.syncCreatedBatch
      -> 镜像失败时尝试删除主库 roast batch
  -> 保存本地批次缓存
```

失败回滚：

| 失败位置 | 回滚策略 |
| --- | --- |
| 创建主记录失败 | 回补生豆库存 |
| 库存回滚也失败 | 抛出需要人工核对的 AppError |
| 熟豆镜像失败 | 尝试删除主记录；删除失败则抛出主库/熟豆库不一致风险 |
| 页面层捕获失败 | 回滚 optimistic batch 并提示 |

### 10.5 更新与删除记录

更新记录时：

1. 卡片右上角“全部编辑”会打开完整 `RoastBatchDrawer mode="edit"`。
2. 完整编辑抽屉底部统一为“取消 / 保存烘焙记录”，并使用保存图标；按钮组位于内容滚动区内，复用 `DrawerActionBar compact` 的 sticky 样式。
3. 单字段编辑继续使用 `RoastBatchFieldEditorDrawer`，底部统一为“取消 / 保存字段名”。
4. 完整编辑抽屉的内容分区、输入间距与四周留白对齐生豆完整编辑抽屉的样式基线。
5. 提交更新前先读取当前 batch。
6. 计算当前库存影响重量与下一状态库存影响重量。
7. 同一生豆只调整差值。
8. 更换生豆时先回补旧生豆，再扣减新生豆。
9. 更新失败时按反向 rollbacks 回滚库存。

删除记录时：

1. 删除入口只保留卡片右上角删除按钮，详情抽屉内不提供删除按钮。
2. 读取当前 batch。
3. 如果记录影响库存，回补生豆库存。
4. 删除 roast_batches。
5. 删除失败时回滚库存调整。

页面 mutation 仍有 Query 缓存层面的乐观移除/恢复。

| 行标签 | 值 |
| --- | --- |
| 全部花费 | `totalExpenses` |
| 已实现收入 | `realizedIncome` |
| 当前库存预估收入 | `estimatedRevenue` |
| 毛利润 | `grossProfit` |
| 经营利润 | `operatingProfit` |

财务口径定义：

1. `全部花费` 只统计实际支出，不统计摊销口径或核算口径成本。
2. `生豆采购` 不通过手工财务台账录入，而是根据生豆记录中的 `采购日期` 与 `购买总价` 自动计入当前统计范围内的支出。
3. `包装`、`邮费` 与其他经营类支出通过“新增支出”手工录入。
4. `已实现收入` 改为按烘焙历史自动汇总：只有去向为“销售”的烘焙记录才计入，且每完成一条销售烘焙记录即按 `1 份 × 默认单份售价` 计入；去向为“自留”的烘焙记录不计入收入。
5. 旧烘焙记录默认按“销售”处理，且后续允许在烘焙历史中改成“自留”或重新改回“销售”。
6. 毛利润按“已经卖出的收入 - 生豆采购 - 包装支出 - 邮费支出”理解。
7. 经营利润按“已经卖出的收入 - 当前时间范围内全部实际支出”理解。
8. `当前库存预估收入` 继续根据剩余生豆、默认售价、默认零售规格和最近脱水率估算。

新增支出抽屉：

1. 字段只保留 `日期 / 类别 / 金额 / 备注`。
2. 类别顺序为 `包装 / 邮费 / 折旧 / 其他 / 自定义`，其中“自定义”固定放在最底部，并允许创建和复用自定义类别。
3. 自定义类别通过历史支出记录沉淀，后续新增支出时继续可选。
4. 备注字段默认不再展示“可留空”辅助文案，仅在校验失败时显示错误提示。
5. 移动端新增支出表单采用扁平化 grouped list 视觉：基础字段为第一组，备注为第二组，弱化卡片感，保持黑白灰与浅分隔线层级。

财务台账同步补充：

1. 手工支出记录远端集合为 `finance_expense_records`。
2. 财务页与全局快速刷新会同时同步 `成本核算 + 财务台账`，并把远端最新记录回填到内存态与 Query Cache。
3. 当 PocketBase 未配置、网络不可用或远端集合尚未建立时，手工支出会直接报错提示，不再降级写入本地持久化缓存。

烘焙历史补充：

1. 烘焙记录新增 `去向` 字段，选项为 `销售 / 自留`。
2. 新建烘焙记录时默认 `销售`。
3. 去向可在烘焙记录完整编辑与字段级编辑中后续修改。
4. 去向展示与保存以 PocketBase 返回结果为准，不依赖本地 bootstrapped 初值或前端乐观缓存先行覆盖。
5. 去向为 `销售` 时，熟豆镜像价格使用生豆中配置的 `默认单份售价`，并参与财务页 `已实现收入` 汇总。
6. 去向为 `自留` 时，熟豆镜像价格按 `（生豆价格 / 1000）× 默认烘焙量` 计算，不参与财务页 `已实现收入` 汇总。

生豆采购补充：

1. 生豆创建和编辑时必须维护 `采购日期`。
2. 生豆采购日期作为财务统计中生豆采购支出的归属日期。
3. 当生豆采购日期或购买总价被修改时，财务统计结果会自动同步变化。
4. 删除生豆后，对应生豆采购支出也会随之从统计结果中消失。
5. 移动端采购日期输入框必须限制在字段容器内，不允许原生日期控件边框突出父级卡片或字段边界。

1. `UnifiedDataCard` 的左侧字段标签保持单行并垂直居中，固定预留至少 6 个汉字宽度；当标签超过该宽度时以省略号截断，不得换行。
2. 左侧标签与右侧字段值至少保留 `12px` 间距；右侧字段值允许按字段配置换行，且不会压缩左侧标签的预留宽度。

烘焙计划删除同步：

1. 用户确认删除后，计划会同时从 Query 缓存和服务层本地镜像中移除。
2. 删除请求进行期间，列表读取和实时同步必须过滤该计划 ID，禁止旧快照回填或将计划重新上传到 PocketBase。
3. 远端删除成功后结束删除状态；只有远端删除失败时才恢复原计划并显示错误提示。
4. PocketBase 是烘焙计划列表的权威数据源；后台同步仅拉取远端计划，不得把“本地存在、远端不存在”的旧计划自动重新上传。
5. 新计划只能通过用户发起的显式创建或 JSON 导入流程上传，不能由后台同步推断创建。
7. 设置页账号栏在移动端与桌面端均需与下方设置卡片保持同一左右边界。

### 11.7 主业务数据服务与缓存处理

主业务数据请求固定通过当前站点同源的 `/api/*` 进入 BFF，不提供普通用户可修改的主库地址。设置页只允许配置熟豆镜像的外部服务；主业务数据缓存由登录状态、写入结果和 realtime 事件统一失效或刷新。

主业务数据服务不可用时，页面不得引导用户创建或连接“生豆数据库”；应提示刷新、检查网络、重新登录或联系管理员。

| 错误 | 行为 |
| --- | --- |

## 17. 当前实现的维护注意点

1. 熟豆 Supabase 专属能力集中在 `RoastedBeanSupabaseDataClient`、`roastedBeanSupabaseConnectionSyncService` 与 `roastedCoffeeBeanMirrorService`；除这两条链路外，主业务数据默认按 PocketBase 理解。
2. `pocketBaseSessionService` 是只含公开用户资料的内存会话，不是持久 localStorage token。刷新后的登录恢复、业务集合访问与 realtime 订阅均通过 HttpOnly Cookie + BFF 完成，前端不得再保存或转发 PocketBase Token。
3. Settings 的连接配置服务目前也是内存状态服务，远端同步通过 app settings 补齐。若做离线持久化，需要谨慎设计迁移和清理策略。
4. 烘焙记录创建、更新、删除会影响生豆库存，是高风险交互；调整前必须补充库存回滚测试。
5. 熟豆镜像失败时有回滚主记录逻辑；如果未来改成最终一致性，需要明确用户可见状态和补偿任务。
6. Finance 已落独立交互页面，但真实销售订单与财务自动汇总仍未接入；当前收入、经营费用属于前端本地台账，后续扩展时应继续保持领域拆分，不要把业务逻辑塞回现有页面。
7. 页面文件中已有一些中文注释和业务流程注释，新增代码仍应保持组件单一职责，避免页面继续膨胀。`BeanPage` 只协调查询、列表、详情编辑与乐观写入；新增方式选择、AI 额度读取和创建抽屉由 `BeanCreationFlow` 维护。`SettingsPage` 只协调远端连接初始化、二维码支持与账号注销；外观和卡片展示设置由 `AppearanceSettingsSection` 维护。页面与组件文件不得超过 500 行。
8. AI 图片识别只负责生成待确认草稿，不直接创建生豆；后续扩展多图识别或字段置信度时，仍应保留用户确认步骤。
9. `deploy.sh` 会发布前端和 BFF：BFF 目标、远端入口文件路径、构建目录、服务名和 SSH 主机可分别通过 `BFF_REMOTE_TARGET`、`BFF_REMOTE_PATH`、`BFF_REMOTE_DIR`、`BFF_SERVICE_NAME`、`REMOTE_SSH_TARGET` 覆盖；前端版本目录、入口链接和发布 SSH 主机可通过 `FRONTEND_RELEASES_PATH`、`FRONTEND_CURRENT_LINK`、`FRONTEND_REMOTE_SSH_TARGET` 覆盖。它先验证和回滚 BFF，再原子发布和回滚前端；脚本会在服务器本机和公网分别验证无凭据 `POST /api/auth/login` 返回 `400`，携带 Cookie 的业务集合请求仍应在发布后按验收步骤抽测。
10. 前端发布目录由 `FRONTEND_RELEASES_TO_KEEP` 控制，默认总数为 `5`，且不得小于 `2`。当前入口和上一版本入口指向的目录永久受保护；公网验收通过后，脚本仅删除超出该总数的最旧非保护版本。当前入口切换、回退入口以及清理均以服务器符号链接解析结果为准，避免删除可回退版本。
11. 发布流程在部署 BFF 前必须获取服务器范围的目录锁，默认路径为 `${FRONTEND_RELEASES_PATH}/.easybake-deploy.lock`。锁由原子创建目录获得，保存本次发布的 owner、创建时间、最近更新时间和执行阶段；锁已存在时后发任务必须立即退出并展示这些元数据，不能上传或切换任何 BFF、前端版本或入口。BFF 部署、前端发布、前端清理三个阶段均刷新锁状态。成功、失败和异常退出均只释放 owner 匹配的锁；前端版本清理必须排除隐藏锁目录。发布客户端强制终止后，必须人工确认任务停止并读取锁元数据，才可删除该精确锁目录；不得自动按超时抢占。
12. 主布局拆分为编排层和展示层：`MainLayout` 只保留路由动画、移动端设置浮层时序、浮动按钮注册、滑动导航和刷新/PWA 状态编排；桌面侧栏、移动顶栏、移动底栏、设置浮层、认证栏、浮动按钮容器和路由面板渲染均放在 `src/layouts/components/`。后续调整导航或设置账号栏时优先修改对应小组件，避免把 JSX 和业务处理重新塞回主布局。
13. 黑白灰规范下，持续可见的结构性强调不再使用绿、蓝、橙、红色条或偏色描边，统一改为灰阶层级表达；只有真正的危险操作与系统级错误可继续使用 danger 语义色。下拉刷新、模块占位页统计卡片、底部导航激活态均已按此规则收口。
14. #28～#33 已完成巨型文件收口：`bean.service.ts`、`roastPlan.service.ts`、`roastBatch.service.ts`、`pocketBaseRestClientCore.ts`、`BeanAiRecognitionPlaceholder.tsx`、`CostTemplateManagerPanel.tsx` 现在都只保留门面编排；后续新增逻辑应优先落到各自的拆分目录（如 `bean-service/`、`roast-plan/`、`roast-batch/`、`pocketbase-rest/`、`bean-ai-recognition/`、`cost-template-manager/`），不要把实现重新堆回主文件。
15. 主布局样式也已按组件归属拆分：壳层滚动与缩放样式集中在 `src/layouts/MainLayoutShell.module.css`；桌面导航、移动头部、底部导航、设置浮层、认证栏、浮动按钮和路由转场各自维护独立 CSS module。后续样式调整优先修改对应组件旁的样式文件，不再回到单一总样式表。

17. `settingsPage.test.tsx` 已保留为主断言文件，但浏览器环境重置、连接卡片展开与运行时 mock 初始化已经抽到 `src/tests/settingsPage.test.shared.tsx`。后续新增设置页测试时，优先复用共享基座，避免测试文件再次膨胀和重复维护。
